## 📖 核心内容

### 一、为什么要评估模型？

**通俗比喻**：评估模型就像"考试"——没有考试，你怎么知道学生学得怎么样？但"考试成绩好"不等于"工作能力强"，这就是模型评估的核心矛盾。

### 二、评估的四个层次

```
┌─────────────────────────────────────────────┐
│  Level 4: 业务价值层                          │
│  (产品指标：留存率、转化率、用户满意度)         │
│  ┌───────────────────────────────────────┐  │
│  │  Level 3: 任务表现层                    │  │
│  │  (特定任务准确率、完成度)               │  │
│  │  ┌─────────────────────────────────┐  │  │
│  │  │  Level 2: 能力维度层             │  │  │
│  │  │  (语言理解、推理、代码、安全)     │  │  │
│  │  │  ┌───────────────────────────┐  │  │  │
│  │  │  │  Level 1: 基准测试层       │  │  │  │
│  │  │  │  (MMLU、GSM8K、HumanEval) │  │  │  │
│  │  │  └───────────────────────────┘  │  │  │
│  │  └─────────────────────────────────┘  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 三、Level 1: 基准测试（Benchmark）

基准测试是学术界的"标准化考试"，用于横向比较不同模型。

| 基准名称 | 测试内容 | 题型 | 满分 | 说明 |
|---------|---------|------|------|------|
| **MMLU** | 多学科知识 | 选择题 | 100 | 覆盖57个学科，是通用能力的主要参考 |
| **CMMLU** | 中文多学科 | 选择题 | 100 | MMLU的中文版，更适合中文模型评估 |
| **GSM8K** | 数学推理 | 问答题 | 100 | 小学数学题，测试逻辑推理能力 |
| **HumanEval** | 代码生成 | 编程题 | 164题 | 测试模型写代码的能力 |
| **TruthfulQA** | 事实准确性 | 问答 | 100 | 测试模型是否会"胡说八道" |
| **C-Eval** | 中文综合能力 | 选择题 | 100 | 涵盖文理工医等多领域 |

**典型模型得分对比**（参考值，非最新）：

| 模型 | MMLU | GSM8K | HumanEval |
|------|------|-------|-----------|
| GPT-4 | 86.4% | 92.0% | 67.0% |
| Claude 3 Opus | 86.8% | 90.7% | 71.1% |
| 文心一言4.0 | 78.5% | 79.3% | 45.2% |
| Qwen-Max | 82.1% | 85.6% | 56.8% |

**产品经理须知**：基准分数≠产品体验！一个模型可能考试分数高，但在你的具体场景中表现可能不如分数低的模型。

### 四、Level 2: 能力维度评估

| 能力维度 | 评估方法 | 关键指标 | 产品经理关注点 |
|---------|---------|---------|---------------|
| **语言理解** | 阅读理解、语义匹配 | 准确率、F1分数 | 能否正确理解用户意图 |
| **语言生成** | 人工评审、BLEU/ROUGE | 流畅度、连贯性 | 回答是否自然、通顺 |
| **逻辑推理** | 数学题、逻辑谜题 | 正确率 | 复杂场景下的推理能力 |
| **代码能力** | 代码生成、调试 | Pass@1（一次通过率） | 辅助编程的实用性 |
| **安全性** | 对抗测试、红队演练 | 有害内容拦截率 | 合规风险、品牌安全 |
| **多语言** | 跨语言翻译、理解 | BLEU、chrF | 国际化产品的语言覆盖 |

### 五、Level 3: 任务表现评估

针对具体产品场景的评估。

| 产品类型 | 评估指标 | 测量方法 |
|---------|---------|---------|
| **智能客服** | 问题解决率、首次解决率 | 用户是否满意、是否需要转人工 |
| **代码助手** | 代码采纳率、修改率 | 开发者接受了多少AI生成的代码 |
| **内容生成** | 人工修改率、可用率 | 生成内容需要多少人工修改 |
| **搜索增强** | 答案相关性、引用准确率 | 答案是否正确、引用是否准确 |

### 六、Level 4: 业务价值评估

最终要回答的问题：**AI功能为业务带来了什么价值？**

| 业务指标 | 含义 | 提升方向 |
|---------|------|---------|
| **用户留存率** | 使用AI功能后是否继续回来 | 体验是否真的好 |
| **任务完成时间** | 用户完成任务所需时间 | 效率是否真的提升 |
| **转化率** | AI推荐/生成是否带来转化 | 建议是否精准有用 |
| **人工替代率** | AI替代了多少人工工作 | ROI是否合理 |
| **用户满意度（CSAT）** | 用户对AI功能的评分 | 整体体验如何 |

### 七、评估方法对比

| 方法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **自动评估** | 速度快、成本低、可规模化 | 不够全面、可能误判 | 快速筛选、大规模对比 |
| **人工评估** | 更准确、能捕捉细微差异 | 成本高、主观性强、速度慢 | 关键功能上线前 |
| **A/B测试** | 真实用户反馈、数据可靠 | 需要足够流量、周期长 | 功能迭代优化 |
| **用户调研** | 了解深层需求、发现盲区 | 样本偏差、回忆偏差 | 新功能探索 |

## 📦 案例分析

### 案例1：GitHub Copilot的效果评估

**评估维度**：

| 指标 | 测量结果 | 业务意义 |
|------|---------|---------|
| **代码采纳率** | ~28%（GitHub官方数据） | 近三成AI建议被直接使用 |
| **开发者效率提升** | 55%开发者表示更快完成任务 | 产品价值得到验证 |
| **用户留存** | 订阅续费率较高 | 开发者依赖该工具 |
| **代码质量** | 混合结果，需人工审查 | AI代码并非总是最优 |

**产品经理启示**：即使采纳率不是100%，只要能显著提升效率，就是成功的产品。

### 案例2：某银行智能客服的评估实践

**背景**：某银行上线AI客服后，需要评估效果。

| 评估阶段 | 方法 | 发现 | 改进 |
|---------|------|------|------|
| 上线前 | 人工评审1000条测试问答 | 金融术语理解不足 | 增加领域微调数据 |
| 上线初期 | A/B测试（50%用户用AI） | AI解决率72%，满意度3.8/5 | 优化复杂问题转人工策略 |
| 上线3个月 | 业务指标分析 | 人工客服工作量下降35% | 扩大AI覆盖场景 |
| 上线6个月 | 用户深度访谈 | 老年用户不适应AI交互 | 增加语音入口、简化交互 |

**产品经理启示**：评估不是一次性动作，而是持续的过程。不同阶段关注不同指标。

---

## 💡 产品经理视角的关键思考

1. **基准分数是参考，不是真理**：不要仅凭MMLU分数选模型，要在自己的场景下实测。
2. **评估要分层**：学术指标、任务指标、业务指标缺一不可。
3. **人工评估不可替代**：尤其是安全、合规、用户体验等方面，自动化评估远远不够。
4. **建立自己的评估集**：收集产品真实用户的问题，构建专属测试集，这才是最有价值的评估依据。

## 📝 思考问题

1. 你的产品使用AI功能来生成产品描述文案。你会设计哪些指标来评估这个功能的效果？请区分"模型指标"和"业务指标"。

2. 为什么有些模型在基准测试中得分很高，但在实际产品中表现却一般？可能的原因有哪些？

3. 如果公司要求你用一个"单一指标"来向老板汇报AI功能的效果，你会选择什么指标？为什么？
