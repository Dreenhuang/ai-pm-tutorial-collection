## 📖 核心内容

### 一、什么是RAG？

**定义**：Retrieval-Augmented Generation，检索增强生成。核心思想是：在模型生成回答之前，先从外部知识库中检索相关信息，然后将检索到的内容作为上下文一起发送给模型。

**通俗比喻**：
- **不用RAG**：让学霸闭卷考试，完全靠记忆回答
- **用RAG**：让学霸开卷考试，可以先查阅资料再回答

### 二、为什么需要RAG？

| 问题 | 纯LLM的局限 | RAG的解决方案 |
|------|------------|--------------|
| **知识过时** | 模型知识截止于训练时间 | 实时检索最新数据 |
| **幻觉问题** | 模型可能编造信息 | 基于检索到的真实内容回答 |
| **私有数据** | 模型不知道企业内部数据 | 接入企业知识库 |
| **可解释性** | 无法追溯信息来源 | 可以提供引用来源 |
| **成本控制** | 大模型训练成本极高 | 只需更新知识库，无需重新训练 |

### 三、RAG架构的核心组件

```
用户问题
    ↓
┌─────────────────┐
│  1. 问题理解     │ ← 可能改写问题、提取关键词
└────────┬────────┘
         ↓
┌─────────────────┐
│  2. 检索模块     │ ← 从知识库找到相关内容
│  (Retriever)    │
└────────┬────────┘
         ↓
┌─────────────────┐
│  3. 内容排序     │ ← 筛选最相关的片段
│  (Ranker)       │
└────────┬────────┘
         ↓
┌─────────────────┐
│  4. 上下文组装   │ ← 将检索内容+问题组合成Prompt
│  (Context)      │
└────────┬────────┘
         ↓
┌─────────────────┐
│  5. 生成模块     │ ← LLM基于上下文生成回答
│  (Generator)    │
└────────┬────────┘
         ↓
    最终回答（可附引用来源）
```

### 四、RAG vs Fine-tuning 对比

| 维度 | RAG | Fine-tuning（微调） |
|------|-----|-------------------|
| **原理** | 检索外部知识辅助生成 | 更新模型权重，内化知识 |
| **数据更新** | 实时更新知识库即可 | 需要重新训练模型 |
| **可解释性** | 可提供引用来源 | 黑盒，无法追溯来源 |
| **幻觉控制** | 较好，基于检索内容回答 | 仍可能产生幻觉 |
| **适用场景** | 知识库问答、文档检索 | 风格迁移、特定任务优化 |
| **成本** | 较低（主要成本在检索） | 较高（需要训练算力） |
| **延迟** | 较高（需要检索+生成） | 较低（直接生成） |

**产品经理决策树**：
```
需要接入私有/实时数据？ → 是 → 选择RAG
                      ↓ 否
需要改变模型行为/风格？ → 是 → 选择Fine-tuning
                      ↓ 否
直接使用基础模型即可
```

### 五、RAG的关键设计决策

| 决策点 | 选项 | 影响 |
|--------|------|------|
| **文档切分** | 按段落/按句子/固定长度 | 影响检索精度 |
| **向量化模型** | OpenAI/本地部署 | 影响成本和效果 |
| **检索策略** | 关键词检索/向量检索/混合 | 影响召回率 |
| **上下文窗口** | 放多少检索内容 | 影响回答质量和成本 |
| **是否重排序** | 简单检索+Rerank | 影响最终相关性 |

### 六、RAG的进阶模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **Naive RAG** | 检索→直接生成 | 简单问答 |
| **Advanced RAG** | 检索→重排序→生成 | 对精度要求高的场景 |
| **Modular RAG** | 可插拔的模块化设计 | 需要灵活组合的场景 |
| **Agentic RAG** | Agent自主决定检索策略 | 复杂多轮对话 |
| **Graph RAG** | 结合知识图谱 | 需要理解实体关系的场景 |

### 七、RAG产品的质量指标

| 指标 | 说明 | 目标值 |
|------|------|--------|
| **检索准确率** | 检索到的内容是否相关 | >80% |
| **回答准确率** | 生成答案是否正确 | >90% |
| **引用准确率** | 引用来源是否匹配答案 | >85% |
| **响应延迟** | 从提问到回答的时间 | <3秒 |
| **幻觉率** | 无依据回答的比例 | <5% |

## 📦 案例分析

### 案例1：企业知识库问答系统

**背景**：某公司有10万份内部文档（政策、流程、产品说明），员工需要快速查询。

**RAG方案设计**：

| 组件 | 方案 | 原因 |
|------|------|------|
| 文档处理 | PDF/Word→文本→分段（500字/段） | 平衡上下文完整性 |
| 向量化 | 使用中文优化的Embedding模型 | 中文检索效果更好 |
| 检索 | 混合检索（关键词+向量） | 提高召回率 |
| 生成 | 基于检索内容生成回答+引用 | 可追溯、减少幻觉 |

**效果**：员工查询时间从平均15分钟缩短到30秒，准确率达92%。

### 案例2：Perplexity AI的搜索体验

**产品分析**：Perplexity是RAG的典型产品化案例。

| 功能 | RAG原理 | 用户体验 |
|------|---------|---------|
| 实时搜索 | 检索最新网页内容 | 回答包含最新信息 |
| 引用来源 | 标注信息来源链接 | 可验证、可追溯 |
| 多源对比 | 检索多个来源综合回答 | 更全面客观 |
| 追问能力 | 基于上文继续检索 | 自然的多轮对话 |

**产品经理启示**：RAG不仅是技术方案，更是一种产品体验范式。

---

## 💡 产品经理视角的关键思考

1. **RAG是解决幻觉问题的实用方案**：但不能完全消除，仍需人工审核关键场景。
2. **知识库质量决定RAG效果**：垃圾数据进入，垃圾回答出来。数据治理是前提。
3. **延迟是用户体验的关键**：检索+生成会增加延迟，需要权衡精度和速度。
4. **引用功能是差异化优势**：能提供引用来源的产品更受企业用户信任。

## 📝 思考问题

1. 你的产品需要接入公司最新的政策法规库（每月更新100+文档），你会选择RAG还是微调？请说明理由。

2. 如果用户对RAG生成的回答提出"这个信息是哪里来的？"，你的产品应该如何设计来回答这个问题？

3. RAG系统中，文档切分策略会影响检索效果。对于一个产品说明书，你会如何设计切分策略？
