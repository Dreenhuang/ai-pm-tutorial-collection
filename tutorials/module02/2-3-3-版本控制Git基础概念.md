## 📖 核心内容

### 一、为什么需要版本控制？

**通俗比喻**：Git就像"时光机+多人协作文档"——它能记录每一次修改（可回退），也能让多人同时编辑同一份文件（不冲突）。

**没有版本控制的问题**：
- "最终版.doc"、"最终版v2.doc"、"真正的最终版.doc"
- 不知道谁改了什么、为什么改
- 改错了无法恢复
- 多人同时修改会互相覆盖

### 二、Git核心概念

| 概念 | 说明 | 通俗理解 |
|------|------|---------|
| **Repository（仓库）** | 存放项目所有代码和修改历史的地方 | 项目的"档案室" |
| **Commit（提交）** | 保存一次代码修改的记录 | 拍一张代码的"快照" |
| **Branch（分支）** | 代码的平行版本 | 故事的"平行宇宙" |
| **Merge（合并）** | 将分支的代码合到一起 | 把平行宇宙合并 |
| **Pull Request（PR）** | 请求别人审核你的代码改动 | "请检查我的修改" |
| **Conflict（冲突）** | 两个人修改了同一行代码 | "这里该听谁的？" |

### 三、Git工作流程

```
日常开发流程：
1. 从远程仓库拉取最新代码（git pull）
2. 创建新分支（git checkout -b feature-login）
3. 修改代码
4. 提交修改（git commit -m "feat: 新增登录功能"）
5. 推送到远程仓库（git push）
6. 创建Pull Request，请求代码审查
7. 审查通过后，合并到主分支
```

### 四、分支策略

| 分支 | 用途 | 命名规范 |
|------|------|---------|
| **main/master** | 主分支，始终是可发布的稳定代码 | main |
| **develop** | 开发分支，集成所有新功能 | develop |
| **feature/** | 功能分支，开发新功能 | feature/user-login |
| **bugfix/** | 修复Bug的分支 | bugfix/fix-login-error |
| **hotfix/** | 紧急修复线上问题的分支 | hotfix/fix-payment-crash |

### 五、常见的Git工作流

| 工作流 | 说明 | 适用场景 |
|--------|------|---------|
| **Git Flow** | 严格的多分支模型 | 大型企业项目 |
| **GitHub Flow** | 简单的main+feature分支 | 互联网产品、持续部署 |
| **GitLab Flow** | 结合Git Flow和GitHub Flow | 需要环境分支的团队 |

## 📦 案例分析

### 案例一：一个功能开发的完整Git流程

**需求**：开发"用户登录"功能

| 步骤 | 操作 | 命令 |
|------|------|------|
| 1. 拉取最新代码 | `git pull origin main` | 确保基于最新代码 |
| 2. 创建功能分支 | `git checkout -b feature/user-login` | 从main创建新分支 |
| 3. 开发功能 | 编写登录页面和接口 | 在feature分支上开发 |
| 4. 提交代码 | `git add . && git commit -m "feat: 新增用户登录功能"` | 保存修改 |
| 5. 推送分支 | `git push origin feature/user-login` | 推送到远程仓库 |
| 6. 创建PR | 在GitHub/GitLab上创建Pull Request | 请求代码审查 |
| 7. 代码审查 | 团队成员Review代码 | 发现问题，提出修改 |
| 8. 合并分支 | 审查通过后合并到main | 功能正式上线 |

## 💡 产品经理视角的关键思考

- 理解Git有助于理解开发的协作流程和排期
- 一个PR通常对应一个需求或Bug修复
- 代码冲突意味着多人修改了同一部分，需要协调
