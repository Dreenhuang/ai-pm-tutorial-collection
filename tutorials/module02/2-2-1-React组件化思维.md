## 📖 核心内容

### 一、React是什么？

**定义**：React是由Facebook开发的用于构建用户界面的JavaScript库，核心思想是"组件化"。

**通俗比喻**：React就像"乐高积木"——把页面拆分成一个个可复用的组件（积木块），通过组合这些组件来构建复杂的界面。

### 二、核心概念

#### 1. 组件（Component）

**定义**：可复用的UI单元，包含自己的结构、样式和逻辑。

```jsx
// 一个按钮组件
function Button({ text, onClick }) {
    return (
        <button className="btn" onClick={onClick}>
            {text}
        </button>
    );
}
```

**组件的特点**：
- 可复用：一个组件可以在多处使用
- 可组合：小组件可以组合成大组件
- 独立维护：修改一个组件不影响其他组件

#### 2. Props（属性）

**定义**：父组件传递给子组件的数据，类似于函数的参数。

```jsx
// 父组件传递
<Button text="点击我" onClick={handleClick} />

// 子组件接收
function Button({ text, onClick }) {
    // text = "点击我"
    // onClick = handleClick函数
}
```

**Props的特点**：
- 单向传递：只能从父到子
- 只读：子组件不能修改Props
- 类型可校验：可规定传递的数据类型

#### 3. State（状态）

**定义**：组件内部的数据，会随着用户交互或API调用而变化。

```jsx
function Counter() {
    const [count, setCount] = useState(0);  // state初始值为0
    
    return (
        <div>
            <p>当前计数: {count}</p>
            <button onClick={() => setCount(count + 1)}>+1</button>
        </div>
    );
}
```

**State的特点**：
- 组件私有：只有组件自己能修改
- 变化触发重渲染：State变化时，UI自动更新
- 可传递给子组件：通过Props向下传递

### 三、React的设计哲学

| 理念 | 说明 | 通俗理解 |
|------|------|---------|
| **声明式** | 描述UI应该是什么样，而不是怎么变 | 说"我要红色"，而不是"把颜色从蓝色改成红色" |
| **组件化** | 界面拆分成独立可复用的部分 | 乐高积木，小组件拼大页面 |
| **单向数据流** | 数据只能从上往下传递 | 水往低处流，不会倒流 |
| **虚拟DOM** | 内存中的UI副本，优化性能 | 先在草稿纸上画好，再画到纸上 |

### 四、React生态系统

| 工具/库 | 用途 | 类比 |
|---------|------|------|
| **React Router** | 页面路由 | 管理不同页面之间的跳转 |
| **Redux/Zustand** | 全局状态管理 | 多个组件共享的"公共数据仓库" |
| **Next.js** | React框架（SSR/SSG） | 让React支持SEO和更快的加载 |
| **Ant Design/MUI** | UI组件库 | 现成的按钮、表格、表单等组件 |
| **React Query** | 数据获取和缓存 | 管理API请求和缓存 |

## 📦 案例分析

### 案例一：电商商品列表的组件化拆解

```
App（整个应用）
├── Header（头部导航）
│   ├── Logo
│   ├── SearchBar
│   └── CartIcon
├── ProductList（商品列表）
│   ├── ProductCard（商品卡片）× N
│   │   ├── ProductImage
│   │   ├── ProductTitle
│   │   ├── ProductPrice
│   │   └── AddToCartButton
│   └── Pagination（分页器）
└── Footer（底部）
```

**组件化思维的好处**：
- ProductCard可以复用到推荐列表、搜索结果、购物车等多个场景
- 修改ProductCard的价格展示样式，所有用到它的地方都自动更新
- 不同团队成员可以并行开发不同组件

### 案例二：状态管理在电商中的应用

**场景**：购物车商品数量变化时，头部导航的购物车图标需要实时更新。

**传统方式**：在多个组件之间手动同步数据，容易出错。

**React方式**：使用全局状态管理（如Redux），购物车数据作为全局State，任何组件都能读取和更新。

## 💡 产品经理视角的关键思考

- 组件化思维也适用于产品设计：将界面拆分为可复用的模块
- 与开发沟通时，用"组件"而非"页面"描述需求更准确
- 理解Props和State有助于理解为什么某些需求改动影响范围大
