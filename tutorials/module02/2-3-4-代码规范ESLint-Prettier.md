## 📖 核心内容

### 一、为什么需要代码规范？

**通俗比喻**：代码规范就像"作文评分标准"——统一格式、统一风格，让每个人的代码读起来像同一个人写的。

**没有代码规范的问题**：
- 同一个项目，有人用2个空格缩进，有人用4个，有人用Tab
- 有人写分号，有人不写
- 变量命名风格不统一（userName、user_name、UserName混用）
- 代码审查时浪费时间在格式争论上

### 二、ESLint vs Prettier

| 维度 | ESLint | Prettier |
|------|--------|----------|
| **作用** | 检查代码质量问题和潜在Bug | 统一代码格式 |
| **关注点** | 代码"对不对" | 代码"好不好看" |
| **示例** | 未使用的变量、可能的错误 | 缩进、引号、分号 |
| **可配置性** | 高（可自定义规则） | 低（"有主见"的格式化） |
| **协作方式** | 两者配合使用 | |

### 三、核心概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **Linting** | 静态代码分析，检查潜在问题 | 发现未使用的变量 |
| **Formatting** | 代码格式化，统一风格 | 统一缩进为2个空格 |
| **规则（Rule）** | 具体的检查或格式化要求 | "必须使用分号" |
| **自动修复** | 工具自动修复部分问题 | Prettier自动格式化 |

### 四、常见规范规则

**ESLint常见规则**：
- `no-unused-vars`：不允许未使用的变量
- `no-console`：不允许使用console.log（生产环境）
- `eqeqeq`：必须使用===而不是==
- `no-debugger`：不允许使用debugger语句

**Prettier常见规则**：
- 缩进：2个空格
- 引号：使用单引号
- 分号：语句末尾加分号
- 行宽：每行最多80个字符

### 五、代码规范的价值

| 价值 | 说明 |
|------|------|
| **提高可读性** | 代码风格统一，更容易阅读理解 |
| **减少Bug** | ESLint能发现潜在问题 |
| **提升协作效率** | 减少代码审查中的格式争论 |
| **新人友好** | 新成员快速融入团队代码风格 |

## 📦 案例分析

### 案例一：没有代码规范的团队

**场景**：5人团队，每人有自己的编码风格。

**问题**：
- 代码审查时，大量时间花在"这个应该用单引号"的争论上
- 新人加入后，不知道该遵循谁的风格
- 合并代码时经常出现格式冲突

**解决方案**：引入ESLint + Prettier，统一规范，提交时自动检查和格式化。

### 案例二：大厂如何管理代码规范？

| 公司 | 规范方式 |
|------|---------|
| 阿里 | ESLint配置（eslint-config-ali） |
| 腾讯 | ESLint + 内部规范 |
| Google | Google JavaScript Style Guide |
| Airbnb | eslint-config-airbnb（被广泛采用） |

## 💡 产品经理视角的关键思考

- 代码规范不是"形式主义"，而是团队协作的基础设施
- 规范越严格，长期维护成本越低
- 与开发沟通时，理解"修lint"是在解决代码质量问题
