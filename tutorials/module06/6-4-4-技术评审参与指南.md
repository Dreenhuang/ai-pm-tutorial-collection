## 📖 核心内容

### 一、什么是技术评审？为什么产品经理要参加？

**通俗比喻**：技术评审就像"装修方案审核"。装修队会告诉你用什么材料、怎么施工。你不需要懂水电布线，但你需要确认"这里放冰箱吗？"、"这个柜子会不会挡住门？"——这些是用户视角的问题。

技术评审的核心目的：

| 目的 | 说明 | 产品经理的价值 |
|------|------|--------------|
| 技术可行性验证 | 方案能不能实现 | 确认方案满足产品需求 |
| 风险识别 | 提前发现技术难点 | 评估对业务的影响 |
| 方案优化 | 找到更好的实现方式 | 平衡技术成本和用户体验 |
| 团队对齐 | 确保大家对方案理解一致 | 消除信息差 |

### 二、技术评审的类型

#### 评审类型对比

| 类型 | 触发时机 | 参与角色 | 产品经理参与度 | 评审重点 |
|------|---------|---------|--------------|---------|
| 架构评审 | 新项目/重大重构 | 架构师、技术负责人 | 中（确认架构支持产品方向） | 技术选型、扩展性、性能 |
| 方案评审 | 具体功能开发前 | 开发、测试、产品 | 高（核心参与者） | 实现方案、影响面、排期 |
| 代码评审 | 代码提交后 | 开发团队 | 低（通常不参与） | 代码质量、规范、潜在Bug |
| 安全评审 | 涉及安全的功能 | 安全工程师、开发 | 中（了解安全风险） | 数据保护、权限控制 |
| 上线评审 | 功能发布前 | 产品、开发、测试、运维 | 高（确认验收通过） | 上线条件、回滚方案 |

### 三、产品经理在技术评审中的角色

#### 参与原则

| 原则 | 含义 | 具体做法 |
|------|------|---------|
| 不干涉技术决策 | 技术怎么做是开发的事 | 关注"能不能实现"，不指导"怎么实现" |
| 聚焦业务价值 | 从用户和业务角度提问 | "这个方案对用户体验有什么影响？" |
| 识别业务风险 | 发现技术方案可能带来的业务问题 | "这个方案会影响现有用户吗？" |
| 促进沟通 | 帮助开发团队理解业务背景 | 补充需求背景和使用场景 |

#### 评审中的关注点

| 关注维度 | 产品经理该问的问题 | 示例 |
|---------|-----------------|------|
| 需求覆盖 | 方案是否覆盖了所有需求点？ | "需求中的XX功能，方案里是怎么实现的？" |
| 用户体验 | 技术方案对用户体验有什么影响？ | "加载时间会增加吗？交互会有变化吗？" |
| 影响范围 | 会影响哪些现有功能？ | "这个改动会影响老用户的数据吗？" |
| 排期合理 | 开发时间预估是否合理？ | "为什么需要2周？主要时间花在哪里？" |
| 风险控制 | 如果出问题，怎么回滚？ | "上线后如果出现问题，回滚方案是什么？" |
| 数据迁移 | 需要迁移数据吗？怎么保证数据不丢失？ | "老数据怎么处理？" |
| 监控指标 | 上线后怎么判断是否成功？ | "上线后看哪些指标？" |

### 四、AI辅助技术评审

#### 1. 评审前：技术方案理解

**提示词模板**：

```
请用非技术人员能理解的语言解释以下技术方案：
技术方案：[粘贴方案文档]

请解释：
1. 这个方案要实现什么？
2. 采用了什么技术？为什么要用这些技术？
3. 对用户有什么影响？
4. 对现有功能有什么影响？
5. 有哪些技术风险？
6. 开发周期预估合理吗？

要求：
- 避免技术术语，或使用比喻解释
- 突出产品经理需要关注的点
- 列出需要确认的问题清单
```

#### 2. 评审前：问题清单准备

**提示词模板**：

```
请基于以下技术方案，为产品经理准备评审问题清单：
需求描述：[描述]
技术方案：[描述]

请准备问题，覆盖以下维度：
1. 需求覆盖度（是否覆盖了所有需求点）
2. 用户体验影响（性能、交互变化）
3. 现有功能影响（兼容性、数据迁移）
4. 排期合理性（时间分配是否合理）
5. 风险控制（上线风险、回滚方案）
6. 监控和验收（如何判断成功）

每个维度准备3-5个具体问题。
输出格式：评审问题清单
```

#### 3. 评审中：实时记录和建议

**提示词模板**：

```
请整理以下评审会议的讨论要点：
会议记录：[粘贴笔记]

请整理为：
1. 达成的共识
2. 待解决的问题
3. 需要确认的信息
4. 后续行动项（谁、做什么、什么时间）
5. 风险点

输出格式：评审会议纪要
```

#### 4. 评审后：方案影响分析

**提示词模板**：

```
请分析以下技术方案变更对产品的影响：
变更内容：[描述]
原方案：[描述]
新方案：[描述]

分析内容：
1. 对用户体验的影响（正面/负面）
2. 对排期的影响（提前/延后）
3. 对需求范围的影响（是否有功能被砍）
4. 对后续迭代的影响
5. 建议的应对措施

输出格式：影响分析报告
```

### 五、技术评审工作流

#### 评审全流程

```
评审前（1-2天）
├── 阅读技术方案
├── AI辅助理解技术内容
├── 准备评审问题清单
└── 与开发提前沟通疑问

评审中（1-2小时）
├── 听取方案讲解
├── 从业务角度提问
├── 记录关键信息和待决事项
└── 确认排期和风险

评审后（当天）
├── AI整理评审会议纪要
├── 跟踪待决事项
├── 更新需求文档（如有变更）
└── 同步相关方评审结论
```

### 六、评审中的提问技巧

#### 提问框架：5W1H

| 维度 | 问题示例 | 目的 |
|------|---------|------|
| What（是什么） | "这个方案具体实现什么功能？" | 确认理解一致 |
| Why（为什么） | "为什么选择这个技术方案？" | 理解决策逻辑 |
| Who（谁） | "这个方案会影响哪些用户？" | 评估影响范围 |
| When（何时） | "什么时候能完成？" | 确认排期 |
| Where（哪里） | "这个功能在哪些页面/场景使用？" | 确认使用范围 |
| How（如何） | "如何保证上线后不出问题？" | 确认风险控制 |

#### 提问禁忌

| 禁忌 | 表现 | 更好的方式 |
|------|------|-----------|
| 质疑动机 | "你为什么选这个方案？"（带质疑语气） | "能帮我理解一下选择这个方案的原因吗？" |
| 外行指导内行 | "我觉得应该用XX技术" | "我担心XX问题，这个方案能解决吗？" |
| 泛泛而问 | "这个方案有什么问题吗？" | "这个方案在数据迁移方面有什么风险？" |
| 现场拍板 | "就按这个方案做" | "我需要回去评估一下影响，明天给你答复" |

### 七、评审意见的分类和处理

| 意见类型 | 说明 | 处理方式 |
|---------|------|---------|
| 必须修改 | 方案存在严重问题，不满足需求 | 要求修改方案，重新评审 |
| 建议优化 | 方案可行，但有改进空间 | 建议开发团队考虑，由开发决定 |
| 信息补充 | 方案信息不完整，需要补充 | 要求补充信息 |
| 风险记录 | 方案可行，但存在已知风险 | 记录风险，制定应对措施 |
| 无异议 | 方案满足需求，无明显问题 | 确认通过 |

### 八、技术评审的常见陷阱

| 陷阱 | 表现 | 避免方法 |
|------|------|---------|
| 过度参与 | 试图指导技术实现 | 明确边界，只关注业务影响 |
| 准备不足 | 评审前不看方案 | 提前阅读，用AI辅助理解 |
| 沉默不语 | 全程不参与讨论 | 准备问题清单，积极提问 |
| 当场拍板 | 没有充分评估就同意 | 复杂方案要求时间评估 |
| 忽略记录 | 评审后不记录结论 | 用AI整理会议纪要，同步相关方 |

## 📦 案例分析

### 案例1：某数据迁移方案的技术评审

**背景**：产品需要进行用户数据迁移，技术方案评审。

**AI辅助评审过程**：
1. AI帮助理解技术方案，发现方案中未提及老用户数据的兼容处理
2. 产品经理在评审中提问："老用户的数据格式和新系统不兼容怎么办？"
3. 开发团队补充了数据转换方案
4. AI整理评审结论，明确数据迁移的验收标准

**结果**：避免了上线后老用户数据丢失的重大事故。

### 案例2：某性能优化方案的评审

**背景**：页面加载慢，开发提出性能优化方案。

**AI辅助评审过程**：
1. AI分析方案，识别出优化可能影响部分老版本浏览器的兼容性
2. 产品经理在评审中确认："这个优化会影响哪些浏览器？我们的用户中有多少用这些浏览器？"
3. 数据反馈显示5%的用户使用受影响的浏览器
4. 团队决定增加兼容处理，确保这部分用户不受影响

**结果**：优化上线后，页面加载速度提升60%，且无用户投诉兼容性问题。

## 💡 产品经理视角的关键思考

1. **技术评审不是"找茬"，而是"共建"**：目标是找到最优方案，而不是证明谁对谁错。
2. **你的价值在业务视角**：技术细节是开发的事，你的价值是确保技术方案不偏离业务目标。
3. **不懂技术不可怕，不懂业务才可怕**：你不需要成为技术专家，但必须成为业务专家。
4. **评审结论要落地**：评审不是走过场，结论必须转化为行动，否则评审没有意义。

## 📝 思考问题

1. 如果技术方案评审中，开发团队的方案与你的预期差异很大，你会如何处理？
2. 你如何判断一个技术方案的风险是"可以接受的"还是"必须解决的"？
3. 当技术评审时间紧张，无法深入讨论所有问题时，你会如何确保关键问题不被遗漏？
