## 📖 核心内容

### 一、什么是需求技术拆解？

**通俗比喻**：需求技术拆解就像把"我想吃一顿大餐"翻译成"需要买菜、洗菜、切菜、炒菜、摆盘"。产品经理告诉厨师"要什么"，技术拆解决定"怎么做"。

需求技术拆解的核心目的：

| 目的 | 说明 | 产品经理的价值 |
|------|------|--------------|
| 让技术可执行 | 将模糊需求转化为具体任务 | 确保技术理解与产品意图一致 |
| 评估工作量 | 为排期和资源分配提供依据 | 合理管理业务方预期 |
| 识别风险 | 提前发现技术难点和依赖 | 主动管理风险，避免延期 |
| 明确验收标准 | 定义什么算"做完了" | 确保交付质量 |

### 二、需求拆解的层次结构

**通俗比喻**：拆解就像剥洋葱，一层一层：

```
用户需求（我想打车）
   ↓
产品需求（一键叫车功能）
   ↓
功能模块（定位、叫车、支付、评价）
   ↓
子功能（地图展示、车辆匹配、价格计算）
   ↓
技术任务（API对接、数据存储、界面开发）
```

#### 各层级的责任分工：

| 层级 | 负责人 | 产品经理参与度 | AI辅助点 |
|------|--------|--------------|---------|
| 用户需求 → 产品需求 | 产品经理 | 主导 | AI辅助需求分析和结构化 |
| 产品需求 → 功能模块 | 产品经理+技术负责人 | 共同 | AI辅助功能拆解和影响分析 |
| 功能模块 → 子功能 | 技术负责人 | 参与评审 | AI辅助任务拆分 |
| 子功能 → 技术任务 | 开发团队 | 了解 | AI辅助工作量评估 |

### 三、AI辅助需求拆解实操

#### 1. 需求功能拆解

**提示词模板**：

```
请将以下产品需求拆解为功能模块和子功能：
需求描述：[描述]
用户场景：[描述]
核心目标：[描述]

拆解要求：
1. 拆分为3-5个功能模块
2. 每个模块拆分为2-4个子功能
3. 标注每个子功能的优先级（P0核心、P1重要、P2一般）
4. 标注子功能之间的依赖关系
5. 标注每个子功能的验收标准

输出格式：功能拆解树+依赖关系表
```

**输出示例**（在线支付功能）：

```
在线支付功能
├── 支付方式模块
│   ├── 支付方式选择（P0）
│   ├── 支付信息展示（P0）
│   └── 支付密码/指纹验证（P0）
├── 支付处理模块
│   ├── 支付请求发起（P0）→ 依赖支付方式选择
│   ├── 第三方支付对接（P0）
│   └── 支付状态轮询（P0）→ 依赖支付请求发起
├── 支付结果模块
│   ├── 支付成功页（P0）→ 依赖支付状态轮询
│   ├── 支付失败页（P1）→ 依赖支付状态轮询
│   └── 支付记录生成（P1）→ 依赖支付成功
└── 异常处理模块
    ├── 网络异常处理（P0）
    ├── 超时处理（P1）
    └── 重复支付防护（P1）
```

#### 2. 需求影响面分析

**提示词模板**：

```
请分析以下需求变更的影响面：
变更内容：[描述]
当前系统状态：[描述相关功能]

分析要求：
1. 直接影响的功能模块
2. 间接影响的关联功能
3. 影响的用户群体
4. 影响的业务流程
5. 需要修改的技术组件（前端、后端、数据库等）
6. 需要回归测试的范围

输出格式：影响面分析报告
```

#### 3. 需求-技术映射表

**提示词模板**：

```
请为以下需求生成需求-技术映射表：
需求列表：[列出]

映射表包含：
1. 需求编号和描述
2. 对应的功能模块
3. 技术实现要点（前端/后端/数据库）
4. 技术难点和风险
5. 预估工作量（人天）
6. 验收标准
7. 依赖的其他需求或模块

输出格式：需求-技术映射表
```

### 四、常见需求拆解方法

#### 1. 用户故事地图（User Story Mapping）

| 步骤 | 操作 | AI辅助点 |
|------|------|---------|
| 识别用户目标 | 用户来产品里要完成什么 | AI分析用户场景 |
| 梳理用户旅程 | 用户从进入到完成的步骤 | AI生成用户旅程图 |
| 拆解活动 | 每个步骤包含哪些操作 | AI辅助活动拆解 |
| 拆解任务 | 每个操作需要做什么 | AI生成任务列表 |
| 划分版本 | 哪些放MVP，哪些放后续 | AI辅助优先级排序 |

#### 2. 功能点分析法（Function Point Analysis）

| 功能点类型 | 说明 | 示例 |
|-----------|------|------|
| 外部输入（EI） | 用户输入数据到系统 | 提交表单、上传文件 |
| 外部输出（EO） | 系统输出数据给用户 | 报表导出、消息推送 |
| 外部查询（EQ） | 用户查询系统数据 | 搜索、列表查看 |
| 内部逻辑文件（ILF） | 系统内部维护的数据 | 用户信息、订单数据 |
| 外部接口文件（EIF） | 系统引用的外部数据 | 第三方API返回的数据 |

#### 3. 工作分解结构（WBS）

**提示词模板**：

```
请为以下项目创建工作分解结构（WBS）：
项目：[描述]
目标：[描述]
周期：[时间]

要求：
- 分解到3级（项目→阶段→任务）
- 每个任务可独立执行和验收
- 标注任务负责人角色
- 标注任务之间的依赖关系
- 标注预估工时

输出格式：WBS树形结构+任务列表表格
```

### 五、需求拆解的验收标准定义

#### 验收标准模板

**提示词模板**：

```
请为以下功能定义验收标准：
功能：[描述]
用户场景：[描述]

验收标准应包含：
1. 功能完整性（核心功能是否可用）
2. 用户体验（交互是否流畅，提示是否清晰）
3. 性能要求（响应时间、并发能力）
4. 兼容性要求（设备、浏览器、系统版本）
5. 异常处理（各种异常情况的处理）
6. 数据安全（权限控制、数据保护）

格式：Given-When-Then（给定什么条件，当做什么操作，然后期望什么结果）
```

**输出示例**：

```
功能：用户登录

1. 功能完整性
   - Given 用户输入正确的账号和密码
   - When 点击"登录"按钮
   - Then 跳转到首页，显示用户信息

2. 异常处理
   - Given 用户输入错误的密码
   - When 点击"登录"按钮
   - Then 提示"账号或密码错误"，停留在登录页

3. 性能要求
   - Given 用户点击"登录"按钮
   - Then 页面响应时间不超过2秒
```

### 六、需求拆解的质量检查

| 检查项 | 说明 | 检查方法 |
|--------|------|---------|
| 完整性 | 是否覆盖所有需求点 | 对照PRD逐项检查 |
| 独立性 | 每个任务是否可独立执行 | 检查任务描述是否清晰 |
| 可测试性 | 验收标准是否可验证 | 每个标准是否可量化 |
| 无歧义 | 描述是否清晰无歧义 | 让非原作者阅读理解 |
| 优先级明确 | 是否标注了优先级 | 检查每个任务是否有优先级 |
| 依赖清晰 | 任务间的依赖是否明确 | 画出依赖关系图 |

## 📦 案例分析

### 案例1：某社交产品的"朋友圈"功能拆解

**背景**：产品规划了类似微信朋友圈的功能。

**AI辅助拆解过程**：
1. AI将需求拆解为6个功能模块：内容发布、内容浏览、互动功能、隐私设置、通知系统、数据管理
2. 影响面分析发现"通知系统"需要复用已有的消息推送模块
3. 验收标准定义中，明确了"发布图文"的性能要求（上传不超过5秒）
4. 技术团队基于拆解结果，预估工作量3周

**结果**：实际开发周期2.5周，比预估提前，因为拆解清晰减少了沟通成本。

### 案例2：某电商平台的"优惠券"功能重构

**背景**：老优惠券系统问题频发，需要重构。

**AI辅助拆解过程**：
1. AI分析老系统的问题，确定重构范围
2. AI拆解新系统的功能模块，识别出"优惠计算引擎"是核心难点
3. 影响面分析发现重构会影响下单流程的3个关联模块
4. 制定分步上线策略：先后台管理，后用户端

**结果**：分步重构降低了风险，后台管理先行让运营团队提前熟悉。

## 💡 产品经理视角的关键思考

1. **拆解的深度取决于团队默契**：默契度高的团队不需要拆得太细，新团队则需要更详细的拆解。
2. **技术拆解不是你的职责，但你要能看懂**：你不需要亲自拆解技术任务，但必须能判断拆解是否合理。
3. **验收标准是保护双方的契约**：清晰的验收标准能避免"做完了但不符合预期"的扯皮。
4. **拆解过程中发现的疑问最有价值**：这些疑问往往是你自己也没想清楚的地方。

## 📝 思考问题

1. 如果你收到一个非常模糊的需求（如"提升用户体验"），你会如何与技术团队沟通并完成拆解？
2. 你如何判断技术团队的拆解结果是否合理？如果预估工作量远超你的预期，你会如何处理？
3. 需求拆解过程中，你发现自己之前的需求描述有遗漏，你会如何处理？
