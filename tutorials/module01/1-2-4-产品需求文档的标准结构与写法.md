## 📖 核心内容

### 一、什么是PRD？

**PRD（Product Requirements Document，产品需求文档）**是产品经理向开发团队传达产品需求的核心文档。它告诉团队"做什么"、"为什么做"、"做成什么样"。

**通俗比喻**：PRD就像装修房子的"施工图纸"——你告诉装修工人每个房间要怎么装、用什么材料、什么效果。没有图纸，装修就会变成"边做边改"，成本高且效果差。

### 二、PRD的核心价值

| 价值 | 说明 | 缺失后果 |
|------|------|---------|
| **需求锚点** | 所有开发的依据和测试的标准 | 开发凭感觉做，结果不符合预期 |
| **沟通载体** | 减少口头传达的歧义和遗漏 | 反复沟通、理解偏差 |
| **决策记录** | 记录为什么要这样做 | 人员变动后，不知道当初为什么这样设计 |
| **验收标准** | 测试和验收的依据 | 上线后才发现功能不完整 |

### 三、标准PRD结构

一份完整的PRD通常包含以下模块：

```
1. 文档信息
2. 需求概述
3. 用户场景
4. 功能需求
5. 非功能需求
6. 数据需求
7. 交互说明
8. 运营需求
9. 附录
```

#### 1. 文档信息

| 字段 | 说明 | 示例 |
|------|------|------|
| 文档名称 | 项目名称+文档类型 | XX功能PRD v1.0 |
| 版本号 | 记录修改历史 | v1.0、v1.1 |
| 作者 | 产品经理 | 张三 |
| 创建日期 | 文档创建时间 | 2024-01-15 |
| 修改记录 | 每次修改的内容和原因 | v1.1：新增微信登录场景 |

#### 2. 需求概述

**目的**：让阅读者快速了解这个需求的背景和目标。

| 字段 | 说明 | 示例 |
|------|------|------|
| 背景 | 为什么要做这个需求？ | 用户反馈登录流程太长，流失率高 |
| 目标 | 希望达到什么效果？ | 降低注册门槛，提升新用户转化率20% |
| 范围 | 做什么、不做什么？ | 做：微信一键登录；不做：其他社交账号登录 |
| 目标用户 | 谁会用到这个功能？ | 未注册的新用户、已有微信的老用户 |
| 优先级 | P0/P1/P2 | P0（最高优先级） |
| 预期上线时间 | 目标上线日期 | 2024-02-01 |

#### 3. 用户场景

用故事化的方式描述用户如何使用这个功能。

**格式**：角色 → 场景 → 目标 → 解决方案

**示例**：

| 场景编号 | 用户角色 | 用户场景 | 用户目标 |
|---------|---------|---------|---------|
| SC-01 | 新用户 | 第一次打开App，想快速体验产品 | 30秒内完成注册并开始使用 |
| SC-02 | 老用户 | 换了手机，需要重新登录 | 通过微信一键登录，无需输入密码 |
| SC-03 | 多账号用户 | 想将微信账号与原账号绑定 | 绑定后可以用两种方式登录 |

#### 4. 功能需求

**核心模块**：用结构化的方式描述每个功能。

**功能描述模板**：

```
功能编号：F-001
功能名称：微信一键登录
功能描述：用户点击微信登录按钮，授权后自动完成注册或登录

前置条件：
1. 用户已安装微信
2. App已接入微信开放平台SDK

主流程：
1. 用户点击"微信登录"按钮
2. 跳转微信授权页面
3. 用户点击"同意授权"
4. 返回App，自动完成登录
5. 跳转到首页

异常流程：
1. 用户取消授权 → 停留在登录页，提示"登录已取消"
2. 未安装微信 → 隐藏微信登录按钮
3. 网络异常 → 提示"网络错误，请重试"
```

#### 5. 非功能需求

| 类型 | 说明 | 示例 |
|------|------|------|
| 性能要求 | 响应时间、并发量 | 登录接口响应时间<1秒 |
| 兼容性要求 | 支持的设备和系统 | iOS 12+、Android 8+ |
| 安全要求 | 数据保护、权限控制 | 用户Token加密传输 |
| 可用性要求 | 系统可用率 | 99.9% |

#### 6. 数据需求

| 字段 | 说明 | 示例 |
|------|------|------|
| 数据埋点 | 需要收集哪些用户行为数据 | 点击微信登录按钮次数、登录成功率 |
| 数据统计 | 需要哪些报表和指标 | 每日新用户注册数、微信登录占比 |
| 数据迁移 | 是否需要处理历史数据 | 老用户绑定微信时需要关联账号 |

#### 7. 交互说明

配合原型图，说明每个交互细节。

| 元素 | 状态 | 交互行为 | 视觉说明 |
|------|------|---------|---------|
| 微信登录按钮 | 默认 | 点击触发授权 | 绿色背景，微信图标 |
| 微信登录按钮 | 加载中 | 显示加载动画 | 按钮变灰，显示loading |
| 登录结果 | 成功 | 跳转首页 | 无弹窗，直接跳转 |
| 登录结果 | 失败 | 显示错误提示 | Toast提示，1秒消失 |

#### 8. 运营需求

| 字段 | 说明 | 示例 |
|------|------|------|
| 上线配置 | 需要配置什么？ | 微信AppID、回调URL |
| 运营工具 | 需要后台管理功能吗？ | 登录数据统计面板 |
| 推广配合 | 需要运营侧做什么？ | 配合上线推送、活动宣传 |

#### 9. 附录

- 竞品分析参考
- 相关文档链接
- 术语解释

### 四、不同场景的PRD详略程度

| 场景 | PRD类型 | 详细程度 | 适用情况 |
|------|---------|---------|---------|
| 全新产品 | 完整PRD | 非常详细（20-50页） | 团队不熟悉产品，需要完整参考 |
| 新功能 | 标准PRD | 中等详细（5-15页） | 常规功能开发 |
| 功能优化 | 轻量PRD | 简要描述（1-3页） | 在现有功能上做小改动 |
| 敏捷迭代 | User Story | 极简（卡片+验收标准） | 团队默契度高，需求简单 |

### 五、PRD的常见格式

| 格式 | 工具 | 优点 | 缺点 |
|------|------|------|------|
| Word文档 | Word | 格式丰富、易于打印 | 版本管理困难、协作不便 |
| 在线文档 | 飞书文档、腾讯文档、Notion | 实时协作、版本管理、评论互动 | 格式相对简单 |
| Wiki系统 | Confluence | 结构化管理、知识沉淀 | 学习成本高 |
| 需求管理工具 | Jira、飞书多维表格 | 与开发流程集成 | 不适合长篇描述 |

**推荐**：使用在线文档（飞书/腾讯文档）作为主要载体，便于协作和版本管理。

## 📦 案例分析

### 案例一：一份优秀的PRD示例（简化版）

**项目**：电商App新增"商品收藏"功能

**1. 需求概述**：
- 背景：用户反馈找不到之前浏览过的商品
- 目标：提升用户复访率，增加商品曝光
- 范围：收藏/取消收藏、收藏列表页

**2. 功能需求**：

```
F-001：收藏商品
- 入口：商品详情页"收藏"按钮
- 流程：点击收藏 → 按钮变为已收藏状态 → 加入购物车附近
- 异常：未登录用户点击 → 引导登录

F-002：收藏列表
- 入口："我的"页面 → "我的收藏"
- 展示：商品图片、名称、价格、收藏时间
- 操作：取消收藏、加入购物车
```

**3. 数据埋点**：
- 收藏按钮点击率
- 收藏列表页访问量
- 收藏商品加购转化率

---

### 案例二：一份失败的PRD

**问题PRD特征**：
1. 只有功能描述，没有背景和目标的说明
2. 没有异常流程，只有"正常路径"
3. 没有数据埋点需求
4. 原型图和文字描述不一致
5. 没有版本管理，不知道最新修改了什么

**改进后**：
1. 每个功能都补充了"为什么做"
2. 增加了完整的异常流程清单
3. 增加了数据埋点和统计需求
4. 原型和文字保持一致
5. 建立版本修改记录

### 案例三：AI产品PRD的特殊要求

**背景**：开发一个AI智能问答功能。

**相比传统PRD的额外要求**：

| 额外模块 | 说明 | 示例 |
|---------|------|------|
| 模型能力说明 | 使用的模型、能力边界 | 基于GPT-4，支持中文问答，不支持图片理解 |
| Prompt设计 | 核心Prompt内容 | "你是一个专业的客服助手..." |
| 效果指标 | 如何衡量AI输出质量 | 回答准确率>90%，用户满意度>4星 |
| Bad Case处理 | 遇到错误输出的处理方案 | 检测到敏感内容时回复默认话术 |
| 人工审核 | 是否需要人工介入 | 低置信度回答转人工客服 |

## 💡 产品经理视角的关键思考

### PRD质量自检清单

**内容完整性**：
- [ ] 是否包含了背景和目标？
- [ ] 是否描述了所有用户场景？
- [ ] 是否列出了正常流程和异常流程？
- [ ] 是否定义了非功能需求？
- [ ] 是否规划了数据埋点？

**表达清晰度**：
- [ ] 开发能否只看文档就理解需求？
- [ ] 是否有歧义或模糊的描述？
- [ ] 原型图和文字描述是否一致？

**可行性**：
- [ ] 是否考虑了技术实现的可行性？
- [ ] 排期是否合理？
- [ ] 是否有已知风险？

## 📝 思考问题

1. 如果开发团队说"PRD太长了不看"，你会如何调整你的文档策略？

2. 对比"完整PRD"和"User Story"两种格式，在什么情况下你会选择使用哪一种？

3. 为什么很多团队说"文档写再好开发也不看"？问题的根源是什么？如何解决？

4. 为一个你熟悉的产品功能（如微信朋友圈的点赞功能）写一份简要PRD（不超过2页）。
