## 核心内容

### 一、数据分析工具的生态地图

**通俗比喻：** 数据分析工具就像厨房里的厨具。菜刀（数据采集）、锅（数据存储）、铲子（数据查询）、盘子（数据展示）。产品经理不需要成为厨师（数据分析师），但要知道每样工具的用途，才能指挥团队"做出一桌好菜"。

数据分析工具有四大类别：

| 类别 | 核心功能 | 典型代表 | 产品经理角色 |
|------|----------|----------|-------------|
| **数据采集/埋点** | 记录用户在产品中的行为 | Sensors Data（神策）、GrowingIO、Google Analytics、友盟 | 定义"要采什么数据" |
| **数据存储/数仓** | 把采集的数据存起来、结构化 | ClickHouse、Snowflake、BigQuery | 不直接操作，但要理解存储限制 |
| **数据查询/分析** | 用 SQL 或可视化界面查数据 | Mixpanel、Amplitude、神策分析 | 日常核心工具，自己做查询 |
| **数据可视化/看板** | 把数据变成图表，给决策者看 | Tableau、Power BI、DataStudio、Grafana | 定义"要看什么图" |

### 二、埋点：数据采集的基石

#### 什么是埋点？

**通俗比喻：** 埋点就像在超市里装摄像头和计数器。你在哪个货架前停留了多久、拿起又放下了哪个商品、最终在收银台花了多少钱——这些数据就是"埋点"采集的。

**定义：** 埋点（Event Tracking）是在产品代码中插入数据采集代码，用于记录用户的特定行为。

#### 埋点事件模型

一个标准的埋点事件包含四个核心字段：

| 字段 | 含义 | 示例 |
|------|------|------|
| **Who** | 谁做的 | 用户 ID：u_12345 |
| **When** | 什么时候 | 2024-03-15 14:30:22 |
| **What** | 做了什么 | 事件：点击"加入购物车" |
| **Where** | 在哪里做的 | 页面：商品详情页 |

此外，还可以有**属性（Properties）**，用来补充上下文信息：

```
事件：购买完成
属性：
  - 商品名称：iPhone 15
  - 商品类目：手机
  - 支付金额：5999
  - 支付方式：支付宝
  - 优惠券：满5000减300
  - 用户来源：抖音广告
```

#### 产品经理如何写埋点需求？

**常见误区：** 产品经理说"把所有能采集的数据都采了"。这是灾难性的——数据太多反而找不到重点，还会拖慢 App 性能。

**正确做法：** 用"事件-属性"表格明确每个埋点：

| 事件名称 | 触发时机 | 必传属性 | 可选属性 | 优先级 |
|----------|----------|----------|----------|--------|
| page_view（页面浏览） | 页面加载完成 | page_name, page_id | source_channel | P0 |
| button_click（按钮点击） | 按钮被点击 | button_name, page_name | - | P1 |
| add_to_cart（加入购物车） | 点击加入购物车按钮 | product_id, price | quantity | P0 |
| purchase_success（购买成功） | 支付成功回调 | order_id, amount, payment_method | coupon_id | P0 |
| search_submit（搜索提交） | 点击搜索按钮 | keywords | result_count | P1 |

**产品经理视角：** 埋点是"基础设施"，上线前就要规划好。产品上线后发现数据采不到，再补埋点就像装修完房子才发现没留插座——要砸墙重来。

### 三、主流工具对比

#### 1. 国内常用工具

| 工具 | 核心优势 | 适用场景 | 免费版 | 学习成本 |
|------|----------|----------|--------|----------|
| **神策数据** | 国内最成熟，支持私有化部署 | 中大型企业，对数据安全要求高 | 有试用版 | 中 |
| **GrowingIO** | 无埋点（自动采集）+ 可视化分析 | 中小团队快速上手 | 有免费版 | 低 |
| **诸葛 IO** | 用户行为路径分析能力强 | 用户旅程分析 | 有免费版 | 低 |
| **友盟+** | 与阿里生态集成好 | 移动 App 基础统计 | 免费 | 低 |
| **TalkingData** | 移动数据智能 | 游戏、电商行业 | 部分免费 | 中 |

#### 2. 国际常用工具

| 工具 | 核心优势 | 适用场景 | 免费版 | 学习成本 |
|------|----------|----------|--------|----------|
| **Google Analytics 4** | 免费、功能全面、与 Google 生态集成 | 网站分析，出海产品 | 免费 | 中 |
| **Mixpanel** | 事件分析最强大，漏斗/留存功能完善 | 产品分析、增长团队 | 月 10 万事件免费 | 中 |
| **Amplitude** | 行为 Cohort 分析、用户画像 | SaaS、增长团队 | 月 1000 万事件免费 | 中高 |
| **Heap** | 自动埋点，回溯分析 | 不想写埋点代码的团队 | 有免费版 | 低 |

#### 3. 选型决策树

```
你的产品是什么类型？
├── 网站 / H5
│   ├── 预算有限 → Google Analytics 4（免费）
│   └── 预算充足 → Mixpanel / Amplitude
├── 移动 App
│   ├── 国内 → 神策数据 / GrowingIO
│   └── 海外 → Firebase Analytics / Mixpanel
├── 小程序
│   └── 微信官方统计 + 神策数据
└── SaaS / B2B
    └── Mixpanel / Amplitude（强调行为分析）
```

### 四、产品经理必备的 SQL 基础

**通俗比喻：** SQL 就是跟数据库"对话"的语言。你想问数据库"上个月有多少人买了东西"，就用 SQL 提问，它给你一个数字作为回答。

产品经理不需要成为 SQL 专家，但掌握以下 5 个命令，可以解决 80% 的日常需求：

```sql
-- 1. SELECT：我要看什么列
SELECT user_id, name, email FROM users;

-- 2. WHERE：筛选条件
SELECT * FROM orders WHERE amount > 100;

-- 3. COUNT：数一数有多少
SELECT COUNT(*) FROM users WHERE created_at > '2024-01-01';

-- 4. GROUP BY：分组统计
SELECT city, COUNT(*) FROM users GROUP BY city;

-- 5. ORDER BY：排序
SELECT product_name, sales FROM products ORDER BY sales DESC LIMIT 10;
```

**实战案例：** 计算某产品的周留存率

```sql
-- 计算第 1 周注册的用户中，第 2 周还有多少活跃
SELECT
    COUNT(DISTINCT CASE WHEN week = 1 THEN user_id END) AS week1_users,
    COUNT(DISTINCT CASE WHEN week = 2 THEN user_id END) AS week2_users,
    COUNT(DISTINCT CASE WHEN week = 2 THEN user_id END) * 100.0 /
    COUNT(DISTINCT CASE WHEN week = 1 THEN user_id END) AS retention_rate
FROM (
    SELECT user_id,
           CASE
               WHEN DATEDIFF(event_date, '2024-01-01') / 7 = 0 THEN 1
               WHEN DATEDIFF(event_date, '2024-01-01') / 7 = 1 THEN 2
           END AS week
    FROM user_events
    WHERE event_date BETWEEN '2024-01-01' AND '2024-01-21'
) t;
```

### 五、数据隐私与合规

**产品经理必须知道：** 数据采集不是想采就能采的。

| 法规 | 核心要求 | 产品经理需做什么 |
|------|----------|-----------------|
| **个人信息保护法（PIPL）** | 采集个人信息需用户同意 | 产品必须有隐私政策弹窗 |
| **GDPR（欧盟）** | 更严格的数据采集限制 | 出海产品必须有"拒绝 Cookie"选项 |
| **数据安全法** | 重要数据不能出境 | 国内数据必须存在国内服务器 |

**产品经理检查清单：**
- [ ] 产品首次启动是否有隐私政策同意流程？
- [ ] 采集的数据是否都有明确用途？
- [ ] 用户是否有权删除自己的数据？
- [ ] 敏感数据（身份证、手机号）是否加密存储？

## 案例分析

### 案例一：小红书的埋点体系

小红书作为社区 + 电商产品，埋点设计极其精细：

**核心埋点事件：**

| 事件 | 触发场景 | 关键属性 |
|------|----------|----------|
| note_exposure | 笔记出现在屏幕中 | note_id, author_id, position |
| note_click | 点击笔记封面 | note_id, source_page |
| note_like | 点赞笔记 | note_id, author_id |
| note_comment | 评论笔记 | note_id, comment_content |
| note_share | 分享笔记 | note_id, share_channel |
| search_query | 搜索关键词 | keywords, result_count |
| cart_add | 加入购物车 | product_id, price |

**产品经理启示：** 社区产品的核心是"内容消费行为"。小红书不只看"用户打开了 App"，而是追踪"用户看了什么笔记、停留多久、有没有互动"。这些细粒度数据是推荐算法优化的基础。

### 案例二：某 SaaS 产品的工具选型

**背景：** 一家 B2B SaaS 公司，产品为项目管理工具，付费用户 5000 人。

**工具选型过程：**

| 阶段 | 使用工具 | 原因 |
|------|----------|------|
| 初创期（<500 用户） | Google Analytics + 自建数据库 | 免费、够用 |
| 成长期（500-5000 用户） | Mixpanel | 需要深入的用户行为分析、漏斗分析 |
| 成熟期（>5000 付费用户） | Mixpanel + Snowflake + Tableau | 需要数仓支持、定制化看板 |

**产品经理启示：** 工具选型不是一次性的决策。随着产品成长，数据需求会指数级增长。早期用免费工具验证，中期引入专业工具，成熟期构建完整的数据平台。不要在用户只有 100 个的时候花 50 万买企业级数据平台。

## 产品经理视角的关键思考

1. **工具是手段，不是目的。** 很多产品经理陷入"工具控"——买最贵的工具、学最复杂的 SQL，但从来没有用数据回答过一个真正的业务问题。先明确"我要解决什么问题"，再选工具。

2. **埋点先行，分析在后。** 产品上线后最痛苦的事：老板问"上个月新功能的转化率是多少？"，你说"抱歉，我们没有埋这个数据"。埋点规划是产品需求文档的一部分，不是开发完成后的补充工作。

3. **免费工具足够起步。** 初学者不要一开始就追求企业级方案。Google Analytics、Mixpanel 免费版、神策试用版，足够你完成 90% 的学习和验证。先跑通"看数据 -> 发现问题 -> 做优化"这个闭环，再考虑升级工具。

4. **数据安全是红线。** 用户数据泄露是公司级别的灾难。产品经理在设计数据采集流程时，必须考虑隐私合规。如果不确定某个数据该不该采，宁可不采。
