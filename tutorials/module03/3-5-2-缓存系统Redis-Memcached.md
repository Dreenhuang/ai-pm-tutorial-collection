## 📖 核心内容

### 一、缓存是什么？

**定义**：缓存（Cache）是将频繁访问的数据存储在高速存储介质（通常是内存）中，以减少对慢速存储（如数据库）的访问次数。

**通俗比喻**：如果数据库是"图书馆"（书多但找书慢），缓存就是"书桌"（只放常用的书，随手可取）。先去书桌找，找不到再去图书馆。

### 二、为什么需要缓存？

**数据库访问的痛点**：

| 问题 | 说明 | 缓存的解决方案 |
|------|------|----------------|
| **查询慢** | 磁盘I/O速度有限 | 内存读取，速度快100倍+ |
| **并发高** | 数据库连接数有限 | 缓存分担大部分请求 |
| **计算复杂** | 聚合查询耗时 | 预计算结果存入缓存 |
| **数据库压力大** | 高并发导致数据库崩溃 | 缓存作为"缓冲层" |

### 三、Redis vs Memcached

| 对比维度 | Redis | Memcached |
|----------|-------|-----------|
| **数据类型** | String、Hash、List、Set、ZSet等 | 仅String |
| **持久化** | 支持（RDB快照+AOF日志） | 不支持 |
| **集群支持** | 原生支持 | 需借助客户端 |
| **单线程/多线程** | 单线程（6.0+支持多线程I/O） | 多线程 |
| **内存利用率** | 高（支持压缩） | 一般 |
| **适用场景** | 复杂数据结构、持久化需求 | 简单缓存、分布式会话 |
| **社区活跃度** | ★★★★★ | ★★★☆☆ |

### 四、缓存的读写策略

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **Cache Aside** | 先读缓存，未命中读数据库并写入缓存 | 简单、常用 | 数据可能短暂不一致 |
| **Read Through** | 缓存负责从数据库加载数据 | 应用代码简单 | 缓存实现复杂 |
| **Write Through** | 写缓存同时写数据库 | 数据一致性好 | 写性能受影响 |
| **Write Behind** | 先写缓存，异步写数据库 | 写性能极高 | 可能丢数据 |

### 五、缓存的常见问题

| 问题 | 说明 | 解决方案 |
|------|------|----------|
| **缓存穿透** | 查询不存在的数据，每次都穿透到数据库 | 布隆过滤器、缓存空值 |
| **缓存击穿** | 热点数据过期瞬间，大量请求打到数据库 | 互斥锁、永不过期 |
| **缓存雪崩** | 大量缓存同时过期，数据库压力骤增 | 随机过期时间、分级缓存 |
| **数据一致性** | 缓存和数据库数据不一致 | 合理设置过期时间、双写一致性方案 |

### 六、缓存的典型应用场景

| 场景 | 说明 | 推荐方案 |
|------|------|----------|
| **热点数据缓存** | 商品信息、用户资料 | Redis String |
| **会话存储** | 用户登录状态 | Redis Hash |
| **计数器** | 点赞数、浏览量 | Redis INCR |
| **排行榜** | 游戏排行榜、热搜 | Redis ZSet |
| **购物车** | 用户购物车数据 | Redis Hash |
| **分布式锁** | 防止超卖、并发控制 | Redis SET NX |
| **限流** | 接口调用频率限制 | Redis + Lua脚本 |

## 📦 案例分析

### 案例一：电商首页的缓存设计

**需求**：电商首页需要快速加载，包含商品推荐、广告、分类导航等。

**缓存策略**：

| 数据 | 缓存方式 | 过期时间 | 更新方式 |
|------|----------|----------|----------|
| 广告位 | Redis String | 1小时 | 运营后台更新时刷新 |
| 分类导航 | Redis String | 24小时 | 定时刷新 |
| 热门商品 | Redis List | 10分钟 | 定时刷新 |
| 用户个性化推荐 | Redis Hash | 30分钟 | 用户行为触发更新 |

**收益**：
- 首页加载时间从2秒降到200ms
- 数据库QPS（每秒查询率）从5000降到500
- 运营人员更新广告后实时生效

### 案例二：秒杀系统的缓存设计

**需求**：秒杀活动开始时，瞬间涌入10万请求。

**设计方案**：

```
用户请求 → Nginx限流 → Redis预扣库存 → 成功则创建订单 → 异步写数据库
                                    ↓
                              库存不足则返回"已售罄"
```

**关键点**：
1. **库存预热**：活动开始前将库存加载到Redis
2. **原子操作**：使用Redis的DECR命令扣减库存，保证不超卖
3. **限流**：Nginx层限制请求频率，保护后端
4. **异步处理**：订单创建成功后，异步写入数据库

**收益**：
- 单机支持10万+QPS
- 不会超卖（Redis原子操作保证）
- 数据库压力极小

## 💡 产品经理视角的关键思考

- **缓存是性能提升的"捷径"**：大多数性能问题可以通过添加缓存解决。但缓存不是银弹，需要考虑数据一致性和缓存更新策略。
- **用户体验的感知**：缓存直接影响页面加载速度。产品经理应该了解哪些数据可以缓存、缓存多久、如何更新。
- **成本意识**：内存比磁盘贵得多。缓存设计需要平衡"命中率"和"内存成本"。不是所有数据都值得缓存。
- **容错设计**：缓存可能失效、可能宕机。产品设计时要考虑"缓存不可用"时的降级方案。
