## 📖 核心内容

### 一、负载均衡是什么？

**定义**：负载均衡（Load Balancing）是将客户端的请求分发到多个后端服务器，以实现流量均匀分配、提高系统可用性和处理能力。

**通俗比喻**：如果服务器是"收银台"，负载均衡就是"排队引导员"——当只有一个收银台时，队伍会越来越长；引导员将顾客分配到多个收银台，所有人都能更快结账。

### 二、为什么需要负载均衡？

| 问题 | 说明 | 负载均衡的解决方案 |
|------|------|-------------------|
| **单点故障** | 一台服务器挂了，服务就中断 | 多台服务器，一台挂了其他继续工作 |
| **性能瓶颈** | 单台服务器处理能力有限 | 多台服务器共同处理 |
| **扩展困难** | 需要停机才能升级服务器 | 动态添加/移除服务器，无需停机 |
| **流量不均** | 某些服务器忙，某些空闲 | 智能分配请求 |

### 三、Nginx是什么？

**定义**：Nginx是一个高性能的HTTP服务器和反向代理服务器，由俄罗斯开发者Igor Sysoev于2004年创建，目前是互联网上使用最广泛的Web服务器之一。

**Nginx的核心功能**：

| 功能 | 说明 | 产品价值 |
|------|------|----------|
| **反向代理** | 接收客户端请求，转发到后端服务器 | 隐藏后端架构，统一入口 |
| **负载均衡** | 将请求分发到多台服务器 | 提高系统可用性和处理能力 |
| **静态文件服务** | 直接提供图片、CSS、JS等文件 | 减轻后端服务器压力 |
| **SSL终结** | 处理HTTPS加密/解密 | 简化后端架构 |
| **缓存** | 缓存后端响应 | 提高响应速度 |

### 四、常见的负载均衡算法

| 算法 | 说明 | 适用场景 | 类比 |
|------|------|----------|------|
| **轮询（Round Robin）** | 按顺序依次分配 | 服务器性能相同 | 发牌 |
| **加权轮询** | 根据服务器性能分配权重 | 服务器性能不同 | 按能力分配任务 |
| **最少连接** | 分配给当前连接数最少的服务器 | 请求处理时间差异大 | 哪个窗口人少排哪个 |
| **IP Hash** | 根据客户端IP分配 | 需要会话保持 | 固定分配到同一窗口 |
| **随机** | 随机分配 | 简单场景 | 抽签 |

### 五、Nginx负载均衡配置示例

```nginx
upstream backend {
    server 192.168.1.10:8080 weight=3;  # 权重3
    server 192.168.1.11:8080 weight=1;  # 权重1
    server 192.168.1.12:8080 backup;    # 备用服务器
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 六、正向代理与反向代理

| 对比维度 | 正向代理 | 反向代理 |
|----------|----------|----------|
| **代表谁** | 代表客户端 | 代表服务器 |
| **客户端知道后端吗** | 客户端知道自己在用代理 | 客户端不知道后端服务器 |
| **典型场景** | 翻墙、匿名访问 | 负载均衡、安全防护 |
| **类比** | 代购（帮你买东西） | 前台（帮你转接） |

**产品经理需要知道的**：大多数互联网产品使用的是反向代理（Nginx），它隐藏了后端架构细节，对外呈现统一的访问入口。

## 📦 案例分析

### 案例一：电商网站的双11负载均衡

**背景**：某电商网站平时日活10万，双11期间日活达到500万。

**负载均衡方案**：

| 层级 | 方案 | 说明 |
|------|------|------|
| **DNS层** | DNS轮询 | 将流量分配到多个数据中心 |
| **CDN层** | CDN缓存静态资源 | 图片、CSS、JS等由CDN提供 |
| **Nginx层** | 多台Nginx + Keepalived | Nginx负载均衡 + 高可用 |
| **应用层** | 多台应用服务器 | 按业务拆分，独立扩展 |

**收益**：
- 系统可承受10倍流量增长
- 单个服务器故障不影响整体服务
- 动态扩展服务器应对流量洪峰

### 案例二：为什么需要"会话保持"？

**问题**：用户登录后，下次请求被分配到另一台服务器，发现需要重新登录。

**原因**：登录状态存储在服务器的内存中，而不是共享存储。

**解决方案**：

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **IP Hash** | 同一IP总是分配到同一服务器 | 简单，但负载均衡不均 |
| **共享会话** | 会话存储在Redis中 | 灵活，但增加复杂度 |
| **Cookie存储** | 会话信息存储在客户端Cookie | 无状态，但有安全风险 |

## 💡 产品经理视角的关键思考

- **高可用是产品底线**：负载均衡不仅是性能问题，更是可用性问题。没有负载均衡，一台服务器故障就意味着整个产品不可用。
- **弹性扩展能力**：负载均衡让系统可以"水平扩展"——通过增加服务器来应对流量增长，而不是不断升级单台服务器。
- **灰度发布的基础**：负载均衡是实现灰度发布（新版本先对部分用户开放）的基础设施。产品经理需要了解这个能力。
- **成本优化**：通过负载均衡，可以使用多台普通服务器替代一台高性能服务器，降低总体成本。
