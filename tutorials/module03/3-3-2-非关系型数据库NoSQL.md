## 📖 核心内容

### 一、什么是NoSQL？

**定义**：NoSQL（Not Only SQL，不仅仅是SQL）泛指非关系型数据库，不使用传统的表格结构，而是根据数据特性采用不同的存储模型。

**通俗比喻**：如果说关系型数据库是"档案柜"（整齐的表格和抽屉），NoSQL就是"储物间"（不同类型的物品用不同的收纳方式——衣服用衣架、书用书架、杂物用箱子）。

**为什么需要NoSQL？**

| 问题 | 关系型数据库的局限 | NoSQL的解决方案 |
|------|-------------------|-----------------|
| **数据结构变化快** | 修改表结构需要停机 | 灵活的数据模型 |
| **海量数据** | 单机性能瓶颈 | 分布式存储 |
| **高并发读写** | 锁机制限制并发 | 无锁或轻量锁 |
| **非结构化数据** | 难以存储JSON、图片等 | 原生支持 |

### 二、NoSQL的四大类型

| 类型 | 代表产品 | 数据模型 | 类比 | 典型场景 |
|------|----------|----------|------|----------|
| **键值型** | Redis、Memcached | Key-Value（键值对） | 字典/快递柜 | 缓存、会话存储 |
| **文档型** | MongoDB、CouchDB | JSON/BSON文档 | 文件夹里的文档 | 内容管理、日志 |
| **列存储型** | Cassandra、HBase | 列族存储 | 按列归档的表格 | 大数据、时序数据 |
| **图数据库** | Neo4j、JanusGraph | 节点+边（关系） | 社交关系图 | 社交网络、推荐系统 |

### 三、Redis详解

**定义**：Redis是一个开源的内存数据结构存储系统，支持字符串、哈希、列表、集合等多种数据结构。

**核心特性**：

| 特性 | 说明 | 场景 |
|------|------|------|
| **内存存储** | 数据存在内存中，读写极快 | 缓存、计数器 |
| **数据结构丰富** | String、Hash、List、Set、ZSet | 排行榜、购物车 |
| **持久化** | 支持RDB快照和AOF日志 | 数据安全 |
| **发布订阅** | 消息广播机制 | 实时通知 |
| **分布式锁** | 保证多节点互斥 | 防止超卖 |

**产品经理常见应用场景**：

| 场景 | 实现方式 | 说明 |
|------|----------|------|
| **热门商品缓存** | `SET product:1001 "{json}"` | 减少数据库压力 |
| **用户会话** | `SET session:abc123 "{user_data}"` | 分布式登录状态 |
| **点赞计数** | `INCR article:1001:likes` | 原子操作，不会出错 |
| **排行榜** | `ZADD rank 100 "user1"` | 按分数排序 |
| **购物车** | `HSET cart:user1 product:1 2` | Hash结构存储 |

### 四、MongoDB详解

**定义**：MongoDB是一个面向文档的NoSQL数据库，以JSON格式（BSON）存储数据。

**核心特性**：

| 特性 | 说明 | 优势 |
|------|------|------|
| **文档模型** | 以JSON文档存储 | 与前端数据格式一致 |
| **灵活模式** | 不需要预定义表结构 | 快速迭代 |
| **水平扩展** | 原生支持分片 | 轻松扩展数据量 |
| **聚合管道** | 强大的数据处理能力 | 数据分析 |

**MongoDB vs MySQL 数据模型对比**：

| 特性 | MySQL | MongoDB |
|------|-------|---------|
| **数据结构** | 表+行+列 | 集合+文档 |
| **关联查询** | JOIN | 嵌入或引用 |
| **模式** | 固定（需DDL） | 灵活（无需预定义） |
| **事务** | ACID完整支持 | 4.0+支持多文档事务 |
| **扩展方式** | 垂直+分库分表 | 原生水平分片 |

### 五、CAP定理

**定义**：CAP定理指出，分布式系统最多只能同时满足以下三个特性中的两个：

| 特性 | 含义 | 说明 |
|------|------|------|
| **C - 一致性**（Consistency） | 所有节点看到的数据相同 | 数据强一致 |
| **A - 可用性**（Availability） | 每个请求都能得到响应 | 服务不中断 |
| **P - 分区容错**（Partition tolerance） | 网络分区时系统仍能工作 | 网络故障不影响 |

**通俗比喻**：开连锁店——
- **C+P**：所有门店库存一致（每次卖完都要同步），但网络断了就无法销售（牺牲A）
- **A+P**：网络断了也能卖（牺牲C，可能超卖）
- **C+A**：理论上不可能，因为网络总会出问题（P是必须的）

**主流数据库的CAP倾向**：

| 数据库 | 倾向 | 说明 |
|--------|------|------|
| MySQL | CP | 强一致性，网络分区时不可用 |
| MongoDB | CP | 默认强一致性 |
| Cassandra | AP | 高可用，最终一致性 |
| Redis | AP | 高可用，数据可能短暂不一致 |

## 案例分析

### 案例一：电商系统为什么同时使用MySQL和Redis？

**架构设计**：
- **MySQL**：存储订单、用户、商品等核心数据（需要事务保证）
- **Redis**：缓存热门商品信息、用户会话、购物车、秒杀库存

**工作流程**：
1. 用户访问商品详情 → 先查Redis缓存
2. 缓存命中 → 直接返回（毫秒级）
3. 缓存未命中 → 查MySQL → 写入Redis → 返回
4. 用户下单 → 写MySQL（事务保证） → 更新Redis缓存

**收益**：
- 数据库查询压力降低90%
- 页面响应速度从500ms降到50ms
- 秒杀场景下不会超卖（Redis原子操作）

### 案例二：社交产品的数据库选型

**需求**：设计一个社交产品的数据存储方案，需要支持用户关系、动态发布、消息推送。

**设计方案**：

| 数据类型 | 数据库 | 理由 |
|----------|--------|------|
| 用户资料、动态内容 | MongoDB | 文档模型灵活，支持嵌套结构 |
| 用户关系图（关注/粉丝） | Neo4j | 图数据库天然适合关系查询 |
| 用户会话、在线状态 | Redis | 内存存储，读写极快 |
| 消息推送日志 | Cassandra | 时序数据，写入量大 |

## 产品经理视角的关键思考

- **没有最好的数据库，只有最合适的**：关系型和NoSQL各有优势，实际项目往往同时使用多种数据库（多语言持久化）。
- **数据结构决定查询效率**：选择数据库时，先思考"我的数据怎么查"，而不是"哪个数据库更流行"。
- **缓存是性能提升的利器**：大多数性能问题可以通过添加缓存解决。但要考虑缓存一致性、缓存穿透等问题。
- **数据一致性是业务决策**：强一致性（如银行转账）牺牲性能，最终一致性（如点赞数）牺牲准确性。产品经理需要根据业务场景做决策。
