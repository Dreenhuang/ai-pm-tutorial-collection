## 📖 核心内容

### 一、HTTP协议是什么？

**定义**：HTTP（HyperText Transfer Protocol，超文本传输协议）是互联网上应用最广泛的数据传输协议。它规定了客户端和服务器之间如何通信。

**通俗比喻**：HTTP就像你去餐厅点菜的"标准流程"——你说出菜品名称（发送请求），服务员记下单子（传输），厨房做好后服务员把菜端给你（返回响应）。整个流程有标准规范，每个人都知道该怎么说、怎么做。

### 二、HTTP工作原理

```
客户端（浏览器）                              服务器
     |                                         |
     |  --- 请求（Request） ----------------→   |
     |     "我要用户ID为1的用户信息"              |
     |                                         |
     |  ←-- 响应（Response） ----------------   |
     |     {"id": 1, "name": "张三", ...}      |
     |                                         |
```

**一次完整的HTTP通信包含四个步骤**：
1. **建立连接**：客户端和服务器之间建立通信通道（三次握手）
2. **发送请求**：客户端告诉服务器需要什么
3. **返回响应**：服务器处理请求后返回结果
4. **关闭连接**：通信完成后断开（HTTP/1.1支持保持连接复用）

### 三、HTTP请求方法

| 方法 | 含义 | 类比 | 典型用途 |
|------|------|------|----------|
| **GET** | 获取数据 | 查看菜单 | 获取用户列表、查询商品信息 |
| **POST** | 提交数据 | 下单点菜 | 创建新用户、提交订单、登录 |
| **PUT** | 更新数据（完整替换） | 整道菜重做 | 修改用户全部信息 |
| **PATCH** | 更新数据（部分修改） | 只加盐不加醋 | 修改用户手机号 |
| **DELETE** | 删除数据 | 退菜 | 删除订单、取消关注 |

**产品经理需要知道的**：当你写PRD描述"新增用户"功能时，开发会选择POST方法；描述"查询用户列表"时，开发会选择GET方法。了解这些有助于你和开发更准确地沟通。

### 四、HTTP状态码

状态码是服务器告诉你"请求处理结果"的数字代码，分为五大类：

| 状态码范围 | 含义 | 常见状态码 | 说明 |
|------------|------|------------|------|
| **1xx** | 信息性响应 | 100 | 请求已接收，继续处理 |
| **2xx** | 成功 | 200 OK | 请求成功，返回数据 |
| | | 201 Created | 资源创建成功 |
| | | 204 No Content | 处理成功但无返回内容 |
| **3xx** | 重定向 | 301 Moved Permanently | 资源已永久移动到新地址 |
| | | 302 Found | 资源临时移动 |
| **4xx** | 客户端错误 | 400 Bad Request | 请求格式有误 |
| | | 401 Unauthorized | 未认证，需要登录 |
| | | 403 Forbidden | 无权限访问 |
| | | 404 Not Found | 资源不存在 |
| **5xx** | 服务器错误 | 500 Internal Server Error | 服务器内部出错 |
| | | 502 Bad Gateway | 网关错误 |
| | | 503 Service Unavailable | 服务暂时不可用 |

**实用记忆口诀**：
- 2开头：成功了
- 3开头：换个地址
- 4开头：你的问题（客户端）
- 5开头：我的问题（服务器）

### 五、HTTP请求的组成部分

一个完整的HTTP请求包含以下部分：

```
POST /api/v1/users HTTP/1.1          ← 请求行（方法 + 路径 + 协议版本）
Host: example.com                    ← 请求头
Content-Type: application/json
Authorization: Bearer xxxxx

{                                     ← 请求体
  "name": "李四",
  "email": "lisi@example.com"
}
```

| 组成部分 | 说明 | 产品经理关注点 |
|----------|------|----------------|
| **请求行** | 方法和路径 | 确认功能对应的接口地址 |
| **请求头** | 附加信息（认证、语言偏好等） | 了解哪些信息会随请求发送 |
| **请求体** | 实际发送的数据 | 确认提交的数据字段 |

### 六、HTTP响应的组成部分

```
HTTP/1.1 200 OK                     ← 响应行（协议版本 + 状态码 + 描述）
Content-Type: application/json      ← 响应头
Content-Length: 123

{                                    ← 响应体
  "id": 1,
  "name": "张三",
  "created_at": "2024-01-01"
}
```

### 七、HTTP vs HTTPS

| 对比维度 | HTTP | HTTPS |
|----------|------|-------|
| **全称** | HyperText Transfer Protocol | HTTP Secure |
| **安全性** | 数据明文传输，可被截获 | 数据加密传输 |
| **端口** | 80 | 443 |
| **证书** | 不需要 | 需要SSL证书 |
| **浏览器标识** | 无 | 地址栏显示锁图标 |
| **性能** | 较快 | 略慢（加密解密开销） |
| **适用场景** | 内部系统、测试环境 | 生产环境、涉及敏感数据 |

**产品经理必须知道**：任何涉及用户隐私、支付、登录的系统必须使用HTTPS。Chrome等浏览器已将HTTP标记为"不安全"，影响用户信任。

## 案例分析

### 案例一：为什么有时网页显示"404 Not Found"？

**现象**：用户点击一个商品链接，页面显示"404 页面不存在"。

**技术原因分析**：
1. 该商品已被下架，数据库中已删除记录
2. 后端收到GET请求后，在数据库中找不到对应商品
3. 后端返回404状态码，告诉浏览器"你要的资源不存在"
4. 浏览器根据404状态码显示错误页面

**产品经理应该做的**：
- 不应该让用户看到冷冰冰的404页面
- 应该设计友好的错误提示页面，引导用户回到首页或推荐相似商品
- 对于下架商品，可以考虑返回301重定向到替代商品

### 案例二：登录时为什么先收到401再跳转？

**现象**：用户访问需要登录的页面时，先被拒绝，然后跳转到登录页。

**流程解析**：
1. 用户直接访问 `/api/v1/orders` （我的订单）
2. 服务器检查请求头中的认证信息（token）
3. 发现没有有效的认证信息，返回 `401 Unauthorized`
4. 前端收到401后，自动跳转到登录页面
5. 用户登录后获得token，再次请求订单页面，返回 `200 OK`

**产品设计启示**：了解这个流程后，产品经理应该设计"未登录状态下的优雅降级"——比如显示部分内容但提示登录查看更多，而不是直接报错。

## 产品经理视角的关键思考

- **理解状态码有助于排查问题**：当用户反馈"系统报错"时，问开发"返回了什么状态码"能快速定位是前端问题、后端问题还是网络问题。
- **设计API需求时要考虑完整性**：不仅要描述正常流程（200），还要描述异常场景（400、401、500等）的交互方式。
- **安全是底线思维**：任何涉及用户数据的功能，必须在PRD中明确要求使用HTTPS，并要求开发说明数据加密方案。
- **性能影响用户体验**：HTTP请求次数直接影响页面加载速度。设计产品时要考虑如何减少不必要的请求（如合并接口、缓存数据）。
