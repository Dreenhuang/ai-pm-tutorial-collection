## 📖 核心内容

### 一、关系型数据库是什么？

**定义**：关系型数据库（Relational Database）是一种以"表"为单位组织数据的数据库系统，表与表之间通过"关系"（外键）关联。

**通俗比喻**：关系型数据库就像Excel表格——每个Sheet是一张表，每行是一条记录，每列是一个字段。不同Sheet之间可以通过ID关联起来。

**核心概念**：

| 概念 | 说明 | 类比 |
|------|------|------|
| **数据库（Database）** | 数据的容器 | 整个Excel文件 |
| **表（Table）** | 存储特定类型数据的结构 | 一个Sheet |
| **行（Row/Record）** | 一条完整的数据记录 | 一行数据 |
| **列（Column/Field）** | 数据的一个属性 | 一列标题 |
| **主键（Primary Key）** | 唯一标识一条记录的字段 | 身份证号 |
| **外键（Foreign Key）** | 关联其他表的字段 | 学号关联学生表 |
| **索引（Index）** | 加速查询的数据结构 | 书的目录 |

### 二、SQL语言基础

SQL（Structured Query Language，结构化查询语言）是操作关系型数据库的标准语言。

**四大操作（CRUD）**：

| 操作 | SQL命令 | 说明 | 示例 |
|------|---------|------|------|
| **创建** | INSERT | 新增数据 | `INSERT INTO users (name, email) VALUES ('张三', 'zhang@example.com')` |
| **查询** | SELECT | 读取数据 | `SELECT * FROM users WHERE age > 18` |
| **更新** | UPDATE | 修改数据 | `UPDATE users SET email = 'new@example.com' WHERE id = 1` |
| **删除** | DELETE | 删除数据 | `DELETE FROM users WHERE id = 1` |

### 三、数据表设计示例

**电商系统的核心表结构**：

**用户表（users）**：
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT (主键) | 用户唯一标识 |
| name | VARCHAR(50) | 用户名 |
| email | VARCHAR(100) | 邮箱 |
| created_at | DATETIME | 创建时间 |

**订单表（orders）**：
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT (主键) | 订单唯一标识 |
| user_id | INT (外键) | 关联用户表 |
| total_amount | DECIMAL(10,2) | 订单金额 |
| status | VARCHAR(20) | 订单状态 |
| created_at | DATETIME | 创建时间 |

**关系**：一个用户可以有多个订单（一对多关系）

### 五、数据库设计原则

| 原则 | 说明 | 违反后果 |
|------|------|----------|
| **范式化** | 消除数据冗余 | 数据不一致 |
| **主键设计** | 使用自增ID或UUID | 数据冲突 |
| **索引优化** | 为常用查询字段建索引 | 查询慢 |
| **字段类型** | 选择合适的数据类型 | 存储空间浪费 |
| **外键约束** | 保证数据完整性 | 脏数据 |

### 六、ACID特性

| 特性 | 含义 | 说明 | 类比 |
|------|------|------|------|
| **A - 原子性** | 事务要么全部成功，要么全部失败 | 转账：扣款和入账必须同时成功 | 要么全做，要么全不做 |
| **C - 一致性** | 事务前后数据保持一致 | 转账前后总金额不变 | 规则不能被破坏 |
| **I - 隔离性** | 多个事务互不干扰 | 两人同时转账不会冲突 | 各做各的 |
| **D - 持久性** | 事务完成后数据永久保存 | 断电后数据不丢失 | 写进去就丢不了 |

## 案例分析

### 案例一：电商系统的订单表设计

**需求**：设计一个电商系统的订单数据结构。

**设计方案**：

```
订单表（orders）
├── 订单基本信息
│   ├── order_id（订单号）
│   ├── user_id（用户ID）
│   ├── status（状态：待付款/已付款/已发货/已完成）
│   └── created_at（创建时间）
├── 支付信息
│   ├── payment_method（支付方式）
│   ├── payment_status（支付状态）
│   └── paid_at（支付时间）
└── 物流信息
    ├── shipping_address（收货地址）
    ├── tracking_number（快递单号）
    └── delivered_at（送达时间）
```

**产品经理关注点**：
- 订单状态流转：必须有明确的状态机设计
- 字段类型：金额用DECIMAL，不用FLOAT（精度问题）
- 时间字段：记录关键时间节点，方便数据分析

### 案例二：为什么"用户表"不能太大？

**问题**：某公司产品有1000万用户，用户表查询越来越慢。

**原因分析**：
1. 单表数据量过大，全表扫描慢
2. 没有合适的索引，查询效率低
3. 字段过多，不常用的字段也放在主表

**优化方案**：
1. **添加索引**：为常用查询字段（如手机号、邮箱）创建索引
2. **分表**：按用户ID范围或注册时间分表
3. **字段拆分**：将不常用字段移到扩展表
4. **读写分离**：主库写，从库读

## 产品经理视角的关键思考

- **数据模型 = 产品骨架**：数据库表结构设计直接影响产品功能的实现难度。产品经理在设计功能时，应该思考"这个功能需要哪些数据字段"。
- **范式化 vs 反范式化**：范式化减少冗余但查询复杂，反范式化提高查询效率但有冗余。实际项目中往往需要权衡。
- **扩展性提前规划**：如果产品预期用户量会快速增长，初期就应该考虑分库分表方案，而不是等问题出现了再补救。
- **数据就是资产**：设计数据表时，要考虑未来数据分析的需求。哪些字段是分析必须的？哪些行为需要记录日志？
