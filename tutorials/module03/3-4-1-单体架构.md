## 📖 核心内容

### 一、单体架构是什么？

**定义**：单体架构（Monolithic Architecture）是指将整个应用的所有功能模块打包成一个独立的单元进行开发、部署和运行。

**通俗比喻**：单体架构就像"瑞士军刀"——所有工具（刀、剪刀、开瓶器）都集成在一个手柄里。你可以用它做所有事情，但如果刀坏了，整个瑞士军刀可能都不能用。

### 二、单体架构的核心特征

| 特征 | 说明 | 产品影响 |
|------|------|----------|
| **单一代码库** | 所有功能在一个代码仓库中 | 代码管理简单，但随着增长变得混乱 |
| **统一构建** | 一次构建生成一个可部署单元 | 部署简单，但构建时间长 |
| **共享数据库** | 所有模块共用一个数据库 | 数据一致性好，但扩展困难 |
| **统一运行** | 所有功能在同一个进程中运行 | 资源共享，但故障可能蔓延 |

### 三、单体架构的优缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **开发效率** | 初期开发快，调试方便 | 后期代码耦合严重，修改一个功能可能影响其他功能 |
| **部署复杂度** | 一次部署全部功能 | 每次更新都要全量部署，风险高 |
| **扩展性** | 简单垂直扩展（加服务器） | 无法针对特定模块单独扩展 |
| **技术栈** | 统一技术栈，团队学习成本低 | 难以引入新技术 |
| **故障隔离** | 无 | 一个模块崩溃可能导致整个系统不可用 |
| **团队协作** | 小团队高效协作 | 大团队多人修改同一代码库容易冲突 |

### 四、典型单体架构示意图

```
┌───────────────────────────────────────┐
│             单体应用                   │
├──────────┬──────────┬─────────────────┤
│  用户模块 │  订单模块 │   商品模块       │
├──────────┴──────────┴─────────────────┤
│            共享数据库                   │
└───────────────────────────────────────┘
```

### 五、何时应该考虑拆分单体？

**演进信号**：

| 信号 | 表现 | 严重程度 |
|------|------|----------|
| **构建时间过长** | 每次构建超过10分钟 | ★★☆☆☆ |
| **代码冲突频繁** | 每天多次合并冲突 | ★★★☆☆ |
| **发布风险高** | 每次发布都担心影响其他功能 | ★★★★☆ |
| **性能瓶颈明显** | 某个模块占用大量资源但无法独立扩展 | ★★★★★ |
| **团队规模增长** | 超过2个团队在同一代码库工作 | ★★★★☆ |
| **技术债务累积** | 修改一个bug需要理解大量无关代码 | ★★★☆☆ |

### 六、单体架构的生命周期

| 阶段 | 团队规模 | 功能数量 | 建议 |
|------|----------|----------|------|
| **初创期** | 1-5人 | 1-10个 | 单体架构最合适 |
| **成长期** | 5-20人 | 10-50个 | 开始模块化，准备拆分 |
| **成熟期** | 20-100人 | 50+个 | 逐步拆分为微服务 |
| **平台期** | 100+人 | 100+个 | 微服务+平台化 |

## 案例分析

### 案例一：淘宝的单体架构演进史

**2003-2005年（单体阶段）**：
- 淘宝初创期使用PHP单体架构
- 团队规模小，功能简单（商品展示、交易、支付）
- 单体架构开发快、部署简单

**2006-2008年（模块化阶段）**：
- 用户量和商品量爆发式增长
- 代码库变得庞大，构建时间变长
- 开始按业务模块划分代码结构

**2009年至今（微服务阶段）**：
- 拆分为数百个微服务
- 每个团队负责自己的服务
- 独立部署、独立扩展

**启示**：没有永远正确的架构，只有适合当前阶段的架构。单体架构不是"错误"，而是"阶段"。

### 案例二：创业公司为什么应该从单体开始？

**背景**：某创业团队3人，开发一款社区产品。

**选择单体的理由**：
1. **速度至上**：需要快速验证产品可行性，而不是花时间在架构设计上
2. **简单就是美**：单体架构调试方便，问题容易定位
3. **成本低**：只需要一台服务器即可运行
4. **团队小**：3个人不会产生代码冲突问题

**6个月后**：
- 用户增长到1万，系统运行稳定
- 开始考虑拆分，因为订单模块开始需要独立扩展

## 产品经理视角的关键思考

- **架构是演进而来的，不是设计出来的**：很多团队在只有100个用户时就设计"微服务架构"，这是典型的过度设计。先让产品跑起来，再根据实际数据决定是否需要拆分。
- **单体架构不是"落后"**：对于大多数中小产品，单体架构是最佳选择。只有当团队规模、功能复杂度、性能需求达到一定程度时，才需要拆分。
- **拆分成本极高**：从单体拆分为微服务，通常需要重写60%以上的代码。产品经理需要评估：这个拆分带来的收益是否值得投入？
- **关注"演进信号"而非"最佳实践"**：不要盲目追随大厂架构，关注自己的产品是否出现了需要拆分的信号。
