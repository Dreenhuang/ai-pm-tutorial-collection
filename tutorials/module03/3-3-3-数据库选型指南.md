## 📖 核心内容

### 一、数据库选型的核心维度

**选型决策树**：

```
你的产品需要什么？
├── 结构化数据 + 事务支持？ → 关系型数据库
│   ├── 国内互联网 → MySQL
│   ├── 复杂查询/分析 → PostgreSQL
│   └── 金融/大型企业 → Oracle
├── 灵活的数据结构？ → 文档型数据库
│   └── 内容管理/日志 → MongoDB
├── 高速读写/缓存？ → 键值型数据库
│   └── 缓存/会话/计数 → Redis
├── 复杂的关系查询？ → 图数据库
│   └── 社交网络/推荐 → Neo4j
└── 海量时序数据？ → 列存储数据库
    └── 日志/监控 → Cassandra/HBase
```

### 二、关系型数据库选型

| 评估维度 | MySQL | PostgreSQL | SQLite |
|----------|-------|------------|--------|
| **适用场景** | Web应用、互联网产品 | 复杂查询、数据分析、地理信息 | 移动端、嵌入式、小型项目 |
| **并发性能** | 高（读多写少场景优秀） | 高（复杂查询优化好） | 低（单文件设计） |
| **数据类型** | 标准类型 | 丰富（JSON、数组、几何） | 基础类型 |
| **扩展性** | 主从复制、分库分表 | 原生分片、逻辑复制 | 不支持分布式 |
| **运维成本** | 低 | 中等 | 极低 |
| **推荐指数** | ★★★★★ | ★★★★☆ | ★★☆☆☆ |

**选型建议**：
- **创业公司/互联网产品**：首选MySQL，生态最成熟，人才最多
- **数据分析/复杂查询**：考虑PostgreSQL，数据类型丰富
- **移动端/桌面应用**：考虑SQLite，零配置，嵌入使用

### 三、NoSQL数据库选型

| 场景 | 推荐数据库 | 理由 | 替代方案 |
|------|------------|------|----------|
| **缓存** | Redis | 性能最优、数据结构丰富 | Memcached（仅字符串） |
| **会话存储** | Redis | 持久化+过期时间 | Memcached |
| **文档存储** | MongoDB | 灵活、查询能力强 | CouchDB |
| **搜索** | Elasticsearch | 全文搜索、分析 | Solr |
| **时序数据** | InfluxDB | 专为时序设计 | TimescaleDB |
| **图数据** | Neo4j | 图算法丰富 | JanusGraph |
| **宽列存储** | Cassandra | 高可用、水平扩展 | HBase |

### 四、混合数据库架构

现代产品往往同时使用多种数据库：

**典型架构示例**：

| 组件 | 数据库 | 职责 |
|------|--------|------|
| 核心业务数据 | MySQL | 用户、订单、支付等需要事务的数据 |
| 缓存层 | Redis | 热门数据、会话、计数器 |
| 搜索 | Elasticsearch | 商品搜索、日志检索 |
| 日志/监控 | InfluxDB | 系统指标、业务指标 |
| 文件存储 | 对象存储（OSS/S3） | 图片、视频、文档 |

### 五、选型常见陷阱

| 陷阱 | 表现 | 避免方法 |
|------|------|----------|
| **盲目追求新技术** | 用MongoDB做订单存储 | 先明确业务需求，再选技术 |
| **过度设计** | 100个用户就用分库分表 | 根据实际数据量决定 |
| **忽视运维成本** | 选择冷门数据库 | 优先选择有成熟生态的数据库 |
| **不考虑数据迁移** | 后期换数据库成本高 | 前期设计数据访问层，便于替换 |
| **忽略备份策略** | 数据丢失无法恢复 | 制定备份和恢复计划 |

### 六、数据库选型的10个关键问题

| 序号 | 问题 | 影响 |
|------|------|------|
| 1 | 数据量有多大？（当前+3年预期） | 决定是否需要分布式 |
| 2 | 读写比例如何？ | 读多写少考虑读写分离 |
| 3 | 是否需要事务？ | 是→关系型，否→NoSQL |
| 4 | 数据结构是否稳定？ | 变化快→文档型 |
| 5 | 查询复杂度如何？ | 复杂关联→关系型 |
| 6 | 响应时间要求？ | 毫秒级→加缓存 |
| 7 | 可用性要求？ | 99.99%→主从+自动切换 |
| 8 | 团队技术栈？ | 熟悉的技术优先 |
| 9 | 运维能力？ | 弱→选托管服务（RDS） |
| 10 | 预算限制？ | 商业数据库成本高昂 |

## 案例分析

### 案例一：社区产品数据库选型

**产品特征**：
- 用户量：10万（预计1年后100万）
- 数据类型：用户资料、帖子、评论、点赞、关注
- 读写特征：读多写少（10:1）
- 核心功能：帖子搜索、用户推荐、实时通知

**选型方案**：

| 数据类型 | 数据库 | 理由 |
|----------|--------|------|
| 用户、帖子、评论 | MySQL | 核心数据需要事务支持 |
| 点赞、计数 | Redis | 高并发写，原子操作 |
| 帖子搜索 | Elasticsearch | 全文搜索、标签过滤 |
| 用户会话 | Redis | 快速读写 |
| 用户关系（关注） | Redis（Set） | 集合操作方便 |
| 通知消息 | Redis（List） | 队列特性 |

### 案例二：IoT设备数据平台

**产品特征**：
- 设备量：1万台，每分钟上报1次数据
- 日数据量：1440万条
- 数据类型：设备ID、时间戳、传感器数值
- 查询特征：按时间范围查询、聚合统计

**选型方案**：

| 数据类型 | 数据库 | 理由 |
|----------|--------|------|
| 设备信息 | MySQL | 结构化数据，量不大 |
| 传感器数据 | InfluxDB | 时序数据库，专为时序优化 |
| 告警规则 | MySQL | 需要事务和关联查询 |
| 实时缓存 | Redis | 最新设备状态 |

## 产品经理视角的关键思考

- **数据就是产品**：数据库选型直接影响产品功能的上限。选错了数据库，后期改造的成本可能是初期的10倍。
- **不要过度设计，也不要欠技术债**：100个用户不需要分库分表，但1000万用户时没有索引会要命。根据产品阶段选择合适方案。
- **托管服务是趋势**：阿里云RDS、AWS Aurora等托管数据库服务，可以大幅降低运维成本。对于没有专职DBA的团队，托管服务是首选。
- **数据迁移是最后手段**：一旦数据存储方案确定，迁移成本极高。选型时要考虑3-5年的业务发展，预留扩展空间。
