## 📖 核心内容

### 一、API网关是什么？

**定义**：API网关（API Gateway）是系统的统一入口，负责接收所有客户端请求，进行路由、认证、限流、监控等处理后，转发到相应的后端服务。

**通俗比喻**：如果后端服务是"公司的各个部门"，API网关就是"公司前台"——所有来访者都先到前台，前台登记身份、判断来意、引导到对应部门。

### 二、API网关 vs 负载均衡

| 对比维度 | 负载均衡（Nginx） | API网关 |
|----------|-------------------|---------|
| **工作层级** | 网络层/传输层 | 应用层（HTTP/REST） |
| **核心功能** | 流量分发 | 流量管理 + 业务处理 |
| **路由方式** | 基于URL/IP | 基于API路径、方法、头部 |
| **认证鉴权** | 简单 | 复杂（JWT、OAuth、API Key） |
| **限流策略** | 基础 | 精细（按用户、API、时间） |
| **数据转换** | 无 | 支持（格式转换、协议转换） |
| **监控分析** | 基础日志 | 详细API指标 |
| **典型产品** | Nginx、HAProxy | Kong、APISIX、Zuul |

### 三、API网关的核心功能

| 功能 | 说明 | 产品价值 |
|------|------|----------|
| **路由转发** | 根据请求路径转发到对应服务 | 统一入口，简化客户端 |
| **认证鉴权** | 验证用户身份和权限 | 集中管理安全策略 |
| **限流熔断** | 限制请求频率，防止系统过载 | 保护后端服务 |
| **缓存** | 缓存常用响应 | 提高响应速度 |
| **日志监控** | 记录所有API调用 | 问题排查和数据分析 |
| **API版本管理** | 支持多版本API并存 | 平滑升级 |
| **数据转换** | 请求/响应格式转换 | 适配不同客户端 |
| **聚合响应** | 合并多个服务的响应 | 减少客户端请求次数 |

### 六、API网关的适用场景

| 场景 | 适合程度 | 说明 |
|------|----------|------|
| **微服务架构** | ★★★★★ | 统一入口，服务路由 |
| **多端适配** | ★★★★★ | 不同端需要不同API格式 |
| **开放平台** | ★★★★★ | 第三方API管理 |
| **API计费** | ★★★★☆ | 按调用次数计费 |
| **单体架构** | ★★☆☆☆ | 功能过剩，增加复杂度 |

## 📦 案例分析

### 案例一：某开放平台的API网关设计

**背景**：某公司开放平台，为第三方开发者提供API。

**API网关承担的职责**：

| 功能 | 实现 | 说明 |
|------|------|------|
| **API Key认证** | 每个开发者分配唯一Key | 身份识别 |
| **限流** | 免费版100次/分钟，付费版1000次/分钟 | 防止滥用 |
| **计费** | 按调用次数计费 | 商业化 |
| **文档** | 自动生成API文档 | 降低开发者接入成本 |
| **监控** | 实时统计调用量、错误率 | 运营分析 |
| **版本管理** | /v1/、/v2/路径区分 | 平滑升级 |

**收益**：
- 统一管理所有API，降低维护成本
- 限流保护后端服务不被恶意调用
- 计费和监控为商业化提供数据支撑

### 案例二：为什么移动端需要API网关？

**问题**：App v1.0和v2.0需要调用不同的API，但后端服务只有一套。

**API网关的解决方案**：

```
App v1.0 → 请求 /api/v1/users → 网关路由到旧版服务
App v2.0 → 请求 /api/v2/users → 网关路由到新版服务
```

**收益**：
- 后端服务可以独立升级
- 老用户不受影响
- 新版本可以灰度发布

## 💡 产品经理视角的关键思考

- **API网关是产品能力的"放大器"**：好的API网关可以让产品更容易被集成、更容易商业化。产品经理应该把API网关视为产品能力的一部分，而不是纯技术基础设施。
- **开放平台的基础设施**：如果产品计划开放API给第三方，API网关是必须的。它提供认证、限流、计费、监控等商业化必需的能力。
- **版本管理是产品决策**：API版本的兼容性直接影响用户体验。产品经理需要决定：新版本是否兼容旧版本？老版本何时下线？
- **性能与功能的权衡**：API网关每增加一个功能（认证、限流、日志），都会增加请求延迟。产品经理需要评估哪些功能是必须的，哪些可以后期添加。
